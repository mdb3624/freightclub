# EC-301 — GPS Heartbeat Monitoring & Carrier Offline Alert

**As a** Shipper,
**I want** to receive an alert when a carrier on an active load stops sending GPS updates for more than 60 minutes,
**so that** I can proactively investigate a potential breakdown, accident, or communication failure before the delivery window is missed.

## Acceptance Criteria
- [ ] `carrier_positions` table stores: `id UUID`, `tenant_id UUID`, `load_id UUID`, `carrier_id UUID`, `position GEOGRAPHY(POINT, 4326)`, `recorded_at TIMESTAMPTZ`; composite index on `(tenant_id, load_id, recorded_at DESC)` for last-position lookups — satisfies **NFR-503**; RLS enforces `tenant_id = app.current_tenant` — satisfies **REQ-103**
- [ ] `HeartbeatMonitorJob` runs every 5 minutes via `@Scheduled`; queries for loads in `IN_TRANSIT` state where `MAX(recorded_at) < NOW() - INTERVAL '60 minutes'`; each qualifying load triggers a `CarrierOfflineEvent` written to `message_outbox`
- [ ] `CarrierOfflineEvent` payload includes `loadId`, `tenantId`, `carrierId`, `lastSeenAt`, and `minutesOffline`; the relay (INF-202) dispatches it as an in-app notification to the shipper
- [ ] A carrier that resumes sending GPS updates after an offline period automatically triggers a `CarrierOnlineRestored` event; the offline alert is marked resolved in `carrier_offline_alerts` table
- [ ] Cyclomatic complexity ≤ 10 in `HeartbeatMonitorJob.checkForOfflineCarriers()` — satisfies **NFR-501**; batch size is capped at 100 loads per cycle to avoid query timeouts
- [ ] `carrier_offline_alerts` table has `tenant_id` column with RLS; soft-delete pattern (`deleted_at IS NULL`) applied — satisfies **REQ-103**
- [ ] JaCoCo branch coverage ≥ 80% for: position received within 60min → no alert; position overdue → alert fired; carrier resumes → alert resolved — satisfies **NFR-502**
