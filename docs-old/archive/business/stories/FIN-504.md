# FIN-504 — Payment Dispute Flow & Settlement Hold

**As a** Shipper,
**I want** to flag a delivery for dispute and place payment on hold,
**so that** I am not charged for a load where the delivery conditions were not met (e.g., damaged goods, wrong drop-off) until the dispute is resolved.

## Acceptance Criteria
- [ ] `POST /api/v1/invoices/{invoiceId}/dispute` (requires `ROLE_SHIPPER`) transitions `invoices.status` from `'UNPAID'` to `'DISPUTED'`; requires a mandatory `reason` field (min 20 characters); payment processing (FIN-503) is blocked while `status = 'DISPUTED'`
- [ ] `payment_disputes` table: `id UUID`, `tenant_id UUID`, `invoice_id UUID`, `load_id UUID`, `raised_by UUID`, `reason TEXT NOT NULL`, `status VARCHAR DEFAULT 'OPEN'`, `resolution TEXT NULL`, `opened_at TIMESTAMPTZ`, `resolved_at TIMESTAMPTZ NULL`; RLS on `tenant_id`; index on `(tenant_id, status)` — satisfies **REQ-103** and **NFR-503**
- [ ] Admin resolves the dispute via `PATCH /api/v1/admin/disputes/{id}` with `{ action: 'approve' | 'reject', resolution: string }`; `approve` → invoice returns to `'UNPAID'` and payment proceeds; `reject` → invoice transitions to `'DISPUTE_REJECTED'` and shipper is notified
- [ ] A `DisputeRaisedEvent` and `DisputeResolvedEvent` are written to `message_outbox` (for notifications) and `audit_log` — satisfies **REQ-402**
- [ ] Both shipper and trucker receive in-app notifications at each dispute status change; dispatched via INF-202 outbox relay
- [ ] Cyclomatic complexity ≤ 10 in `DisputeService.raise()` and `DisputeService.resolve()` — satisfies **NFR-501**
- [ ] Disputes cannot be raised on loads in `SETTLED` status (payment already confirmed); attempting returns `409 Conflict` with `LOAD_ALREADY_SETTLED`
- [ ] JaCoCo branch coverage ≥ 80% for: valid dispute → DISPUTED; admin approve → UNPAID; admin reject → DISPUTE_REJECTED; dispute on settled load → 409 — satisfies **NFR-502**
