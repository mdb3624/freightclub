# MAT-303 — Equipment Compatibility Matrix & Hierarchy Rules

**As the** Match Engine,
**I want** to apply an equipment compatibility matrix when filtering carriers for a load,
**so that** a carrier with a higher-capability trailer (e.g., Reefer) can haul loads requiring a lower-capability type (e.g., Dry Van), but never the reverse.

## Acceptance Criteria
- [ ] `EquipmentCompatibilityMatrix` is a domain-layer singleton (pure Java, no Spring annotations) that encodes the hierarchy: `REEFER ⊇ DRY_VAN ⊇ FLATBED`; specialized types (`HAZMAT`, `OVERSIZED`) are self-matching only — satisfies **REQ-302**
- [ ] `EquipmentCompatibilityMatrix.canCarry(carrierEquipment, loadRequirement)` returns `boolean`; cyclomatic complexity ≤ 10 — satisfies **NFR-501**; implemented as a lookup map, not a chain of if-else
- [ ] `CarrierSearchPort` applies the compatibility filter after the spatial filter (MAT-301) and before the profitability score (MAT-302); filter logic: `matrix.canCarry(carrier.equipmentType, load.requiredEquipmentType)`
- [ ] `loads` table has `required_equipment_type VARCHAR NOT NULL`; `carrier_profiles` has `equipment_type VARCHAR NOT NULL`; both use the same `EquipmentType` enum defined in the shared kernel
- [ ] `EquipmentCompatibilityMatrixTest` (unit test, no Spring context) covers: Reefer carrying Dry Van → `true`; Dry Van carrying Reefer → `false`; Hazmat carrying Dry Van → `false`; same type → `true`; all 16 combinations in the matrix are explicitly asserted — satisfies **NFR-502**
- [ ] Load board UI surfaces the required equipment type badge on each load card; incompatible loads are greyed out (not hidden) so truckers understand why they cannot claim
- [ ] Flyway migration adds `required_equipment_type` column to `loads` with a `NOT NULL` constraint and a `CHECK` constraint limiting values to known enum members
