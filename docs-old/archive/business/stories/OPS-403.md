# OPS-403 — In-Transit Progression: Trip Start, Arrival Confirmation & POD-Gated Delivery

**As a** Trucker (Owner-Operator),
**I want** to advance a claimed load through the remaining states — IN_TRANSIT → ARRIVED → DELIVERED — with each transition properly guarded,
**so that** the shipper receives real-time status visibility and the system prevents premature delivery confirmation without physical proof.

## Acceptance Criteria
- [ ] `LoadAggregate` supports the complete FSM required by **REQ-201**: `DRAFT → PUBLISHED → CLAIMED → IN_TRANSIT → ARRIVED → DELIVERED`; the `ARRIVED` state is a mandatory intermediate step between `IN_TRANSIT` and `DELIVERED` — skipping it throws `LoadDomainException`
- [ ] `PUT /api/v2/loads/{id}/start-trip` transitions `CLAIMED → IN_TRANSIT`; accessible only to `ROLE_TRUCKER` who holds the `carrierId` on the aggregate; attempting with wrong carrier returns `403 Forbidden`
- [ ] `PUT /api/v2/loads/{id}/arrive` transitions `IN_TRANSIT → ARRIVED`; triggers `LoadArrivedEvent` written to `message_outbox`; shipper receives an in-app notification within one relay cycle
- [ ] `PUT /api/v2/loads/{id}/deliver` transitions `ARRIVED → DELIVERED`; requires a valid `pod_url` in the request body AND a `load_documents` row with `document_type = 'POD'` for this load; missing POD returns `422 Unprocessable Entity`
- [ ] Each transition drains `pullDomainEvents()` and writes to `message_outbox` within the same `@Transactional` boundary — satisfies **REQ-202**
- [ ] Flyway migration adds `ARRIVED` to the `load_status` enum or `VARCHAR` column; existing data is not affected (only new rows can enter `ARRIVED`)
- [ ] `LoadAggregateTest` (domain unit test, zero Spring dependencies) covers all valid transitions and every invalid transition (`LoadDomainException` expected); JaCoCo branch coverage ≥ 80% — satisfies **NFR-502**
- [ ] Code must adhere to **NFR-501**: each state-transition method on `LoadAggregate` must not exceed complexity 10
