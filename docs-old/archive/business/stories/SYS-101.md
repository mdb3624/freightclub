# SYS-101 — Tenant USDOT Binding & Authority Verification at Onboarding

**As a** Platform Administrator,
**I want** every new carrier tenant to have their USDOT number verified against the FMCSA SAFER Web registry before they can be activated,
**so that** only carriers with active operating authority are permitted to claim loads, and the platform cannot be used by revoked or fictitious operators.

## Acceptance Criteria
- [ ] `tenant_registrations` table includes `usdot_number VARCHAR(10)`, `authority_status VARCHAR(20)`, `verified_at TIMESTAMPTZ`, and `tenant_id UUID`; RLS enforces `tenant_id = app.current_tenant` — satisfies **REQ-103**
- [ ] On carrier tenant registration, `UsdotOnboardingService` calls the FMCSA SAFER API synchronously; a timeout (≥ 5s) or non-200 response writes a `PENDING_VERIFICATION` status and schedules a retry via `message_outbox` — satisfies **REQ-101**
- [ ] Tenants with `authority_status = 'INACTIVE'` or `authority_status = 'REVOKED'` are blocked from all load-claiming endpoints (`/api/v2/loads/{id}/claim`); the response is `403 Forbidden` with code `AUTHORITY_INACTIVE`
- [ ] Flyway migration creates the `tenant_registrations` table with `ENABLE ROW LEVEL SECURITY` and `FORCE ROW LEVEL SECURITY`; composite index on `(tenant_id, authority_status)` supports status checks
- [ ] `UsdotVerificationResult` is a domain Value Object in `modules/carrier/domain`; it contains no Spring or JPA annotations — satisfies **NFR-501** (complexity ≤ 10 per method)
- [ ] Integration test verifies: valid USDOT → tenant activated; inactive USDOT → tenant blocked; FMCSA timeout → status = PENDING_VERIFICATION with retry scheduled
