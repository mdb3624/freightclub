# US-102: Multi-Tenant Context & JWT Authentication Infrastructure

**Epic:** Module 1 – Onboarding & Identity (The Foundation)  
**Category:** Technical Infrastructure  
**Priority:** Blocker (enables all downstream multi-tenant features)  
**Status:** Backlog

---

## 📖 User Story

**As a** System Architect  
**I want to** establish a thread-safe tenant context holder and integrate it with JWT authentication  
**So that** every HTTP request is bound to a `tenant_id`, enabling Row-Level Security (RLS) policies to enforce tenant isolation automatically across all downstream operations.

---

## 🎯 Acceptance Criteria

### AC-1: TenantContextHolder Thread-Safety
- [ ] `TenantContextHolder` uses `ThreadLocal<String>` to store the current `tenant_id` per request thread
- [ ] All access is synchronized internally; callers do not manage locks
- [ ] Concurrent requests from different tenants do **not** share or leak `tenant_id` values
- [ ] **Test:** Spin up 10 parallel threads, each setting a different `tenant_id`, verify isolation via assertion
- [ ] Reference: [CLAUDE.md - Context Section](CLAUDE.md)

### AC-2: JwtAuthenticationFilter Context Initialization
- [ ] Filter extracts `tenant_id` claim from JWT payload during `doFilterInternal()`
- [ ] `TenantContextHolder.setTenantId(tenantId)` is called before request processing
- [ ] If JWT is missing or invalid, the request is rejected with HTTP 401 Unauthorized
- [ ] If JWT exists but `tenant_id` claim is absent, the request is rejected with HTTP 403 Forbidden
- [ ] **Test:** Verify filter chain propagates `tenant_id` to downstream `@RestController` methods

### AC-3: Context Cleanup in Finally Block
- [ ] JwtAuthenticationFilter implements context cleanup in a `finally` block that always executes
- [ ] `TenantContextHolder.clear()` is called in the `finally` block to prevent ThreadLocal leaks
- [ ] ThreadLocal is cleared **before** the response is sent, even if an exception occurs
- [ ] **Test:** Throw an exception mid-request, verify ThreadLocal is still cleared and no stale context remains
- [ ] Verify two consecutive requests from the same thread get independent contexts (no cross-request bleed)

### AC-4: No-Lombok Policy Compliance
- [ ] Zero Lombok annotations in `TenantContextHolder` or `JwtAuthenticationFilter`
- [ ] Use manual getters/setters, Records, or plain Java POJOs
- [ ] All bean constructors and field declarations are explicit (no `@Data`, `@Getter`, `@Setter`)
- [ ] Code review: Ensure no `import lombok.*` statements exist in the implementation
- [ ] Reference: [PROJECT_CONSTITUTION.md](PROJECT_CONSTITUTION.md)

### AC-5: Spring Security Integration
- [ ] Filter is registered as a Spring Security filter via `SecurityFilterChain` bean
- [ ] Filter executes **before** `UsernamePasswordAuthenticationFilter` to ensure context is available to downstream logic
- [ ] Bean is annotated with `@Component` and `@Order(Ordered.HIGHEST_PRECEDENCE)` or explicitly ordered in `SecurityConfig`
- [ ] **Test:** Verify filter is called in correct order relative to Spring's built-in filters

### AC-6: JWT Claim Validation
- [ ] JWT is decoded using the configured secret key (HS256 algorithm)
- [ ] `tenant_id` claim is validated as a non-empty UUID or string
- [ ] If decoding fails (invalid signature, expired, malformed), request is rejected with 401
- [ ] **Test:** Provide valid JWT with missing `tenant_id`, invalid JWT, expired JWT—all must be rejected

### AC-7: Documentation & Traceability
- [ ] Implementation includes code comments explaining **why** ThreadLocal cleanup is mandatory
- [ ] Exception handling is explicit—no silent failures
- [ ] Error responses include meaningful `WWW-Authenticate` header for 401/403 cases
- [ ] Javadoc on `TenantContextHolder.clear()` warns against manual calls outside filter lifecycle
- [ ] Reference: Verify against [REQUIREMENTS.md](../REQUIREMENTS.md) Multi-Tenancy section

## 🔐 Technical Notes & Constraints

### ThreadLocal Lifecycle
- Context is set at filter entry point
- Cleared in finally block before response leaves filter chain
- Subsequent filters and handlers can read via `TenantContextHolder.getTenantId()`

### Exception Handling
- If filter throws an exception, finally block **must still execute**
- Use try-finally, not try-catch, to avoid swallowing security exceptions
- Spring's `ExceptionHandlerFilter` or `@ControllerAdvice` will handle the response

### Testing Strategy
- **Unit Tests:** Mock JWT decoder, verify context is set and cleared
- **Integration Tests:** Use TestSecurityContext or `@WithMockUser` + custom SecurityContextHolder logic
- **Concurrency Tests:** Spawn 20+ threads with unique `tenant_id` claims, verify isolation

### Future Dependencies
- Once this AC is complete, all repository queries in [REQUIREMENTS.md](../REQUIREMENTS.md) must include explicit `tenant_id` filtering OR rely on PostgreSQL RLS policies
- Load claiming operations (`@Lock` in JPA) depend on this context being available
