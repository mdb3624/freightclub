# DB-003 — Foreign Key: `loads.trucker_id` → `users.id`

**As an** engineer,
**I want** a foreign key constraint from `loads.trucker_id` to `users.id`,
**so that** referential integrity is enforced at the database level and orphaned trucker references are impossible.

## Acceptance Criteria

- [ ] A Flyway migration adds `CONSTRAINT fk_loads_trucker FOREIGN KEY (trucker_id) REFERENCES users(id)`.
- [ ] The constraint uses `ON DELETE SET NULL` (a deleted user leaves the load unclaimed rather than causing a cascade delete).
- [ ] Before adding the constraint, the migration verifies (or cleans) any existing `trucker_id` values that reference non-existent users.
- [ ] After migration, attempting to insert a load with a non-existent `trucker_id` fails with a foreign key violation.
- [ ] `trucker_id` remains nullable (a load without an assigned trucker is valid).
- [ ] Migration follows the Flyway naming convention.
