# DB-004 — Introduce `claims` Table

**As an** engineer,
**I want** a dedicated `claims` table to record claim history per load,
**so that** cancellation notifications, trucker ratings, and future bidding have a reliable, normalized data source rather than a single `trucker_id` column on `loads`.

## Acceptance Criteria

- [ ] A Flyway migration creates a `claims` table with at minimum: `id` (UUID PK), `tenant_id`, `load_id` (FK → loads), `trucker_id` (FK → users), `claimed_at` DATETIME, `released_at` DATETIME NULL, `release_reason` VARCHAR NULL, `status` ENUM('ACTIVE','RELEASED','CANCELLED').
- [ ] `tenant_id` is present on the table for multi-tenancy scoping.
- [ ] Appropriate indexes are added: `(tenant_id, load_id)`, `(tenant_id, trucker_id, status)`.
- [ ] When a trucker claims a load, a row is inserted into `claims`.
- [ ] When a load is cancelled or the trucker is released, the claim row is updated (`released_at`, `release_reason`, `status`).
- [ ] `loads.trucker_id` is retained for now but considered a denormalized cache; it must stay in sync with the active claim.
- [ ] Migration follows the Flyway naming convention.
