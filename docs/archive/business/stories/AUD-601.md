# AUD-601 — Immutable Transaction Ledger & Audit Trail

**As a** Platform Administrator,
**I want** every significant state change — load lifecycle transitions, role assignments, document uploads, and financial requests — recorded in an append-only ledger,
**so that** the platform can satisfy regulatory inquiries, dispute resolutions, and internal forensic reviews without relying on mutable application tables.

## Acceptance Criteria
- [ ] `audit_log` table has columns: `id UUID`, `tenant_id UUID`, `actor_id UUID`, `event_type VARCHAR`, `entity_id UUID`, `payload JSONB`, `occurred_at TIMESTAMPTZ DEFAULT NOW()`; no `UPDATE` or `DELETE` is permitted — enforced by a PostgreSQL trigger that raises an exception on any attempt
- [ ] RLS on `audit_log` allows `SELECT` only where `tenant_id = app.current_tenant`; platform admins have a superuser bypass role for cross-tenant forensic queries
- [ ] `AuditEventPublisher` is called within the same `@Transactional` boundary as the triggering action; if the audit write fails, the entire transaction rolls back
- [ ] Covered events: `LOAD_DRAFTED`, `LOAD_PUBLISHED`, `LOAD_CLAIMED`, `LOAD_CANCELLED`, `LOAD_DELIVERED`, `USER_ROLE_CHANGED`, `DOCUMENT_UPLOADED`, `FACTORING_REQUESTED`
- [ ] `payload` stores the full before/after state as JSONB; PII fields (`email`, `phone`) are replaced with a SHA-256 hash in the stored payload
- [ ] A `GET /api/v1/admin/audit?entityId={id}` endpoint returns paginated log entries; access is restricted to `ROLE_ADMIN` only
- [ ] JaCoCo branch coverage for `AuditEventPublisher` must be ≥ 80%
