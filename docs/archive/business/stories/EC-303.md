# EC-303 — Multi-Stop Rerouting & Rate Re-Confirmation

**As a** Shipper,
**I want** to add a stop to an already-claimed load and trigger a rate re-negotiation with the carrier,
**so that** the carrier is fairly compensated for the additional miles and the load contract reflects the updated route.

## Acceptance Criteria
- [ ] `load_stops` table: `id UUID`, `load_id UUID`, `tenant_id UUID`, `sequence INT`, `address TEXT`, `arrival_window_start TIMESTAMPTZ`, `arrival_window_end TIMESTAMPTZ`, `added_at TIMESTAMPTZ`; RLS on `tenant_id`; composite index on `(tenant_id, load_id, sequence)` — satisfies **REQ-103** and **NFR-503**
- [ ] Adding a stop via `PUT /api/v2/loads/{id}/stops` is only permitted when the load is in `CLAIMED` or `IN_TRANSIT` status; any other status returns `409 Conflict`
- [ ] When a stop is added to a `CLAIMED` or `IN_TRANSIT` load, the `LoadAggregate` emits a `LoadReroutedEvent` and transitions status back to `CLAIMED`; a `RateReConfirmationRequest` is written to `message_outbox` and an in-app notification is sent to the carrier — satisfies **REQ-202**
- [ ] Carrier must explicitly accept the new rate via `PUT /api/v2/loads/{id}/confirm-reroute` within a configurable window (default 30 minutes); if they do not respond, the load reverts to `PUBLISHED` and the claim is released
- [ ] `LoadAggregate.addStop()` domain method enforces: maximum 5 stops per load; sequence numbers must be contiguous; duplicate stop addresses within the same load are rejected
- [ ] Cyclomatic complexity ≤ 10 in `LoadAggregate.addStop()` — satisfies **NFR-501**
- [ ] `LoadReroutedEvent` is recorded in `audit_log` with before/after stop list as JSONB payload — satisfies **REQ-402**
- [ ] JaCoCo branch coverage ≥ 80% covering: valid stop addition → CLAIMED + notification; timeout → released to PUBLISHED; max stops exceeded → rejection — satisfies **NFR-502**
