# EIA-105 — EIA API Health Monitoring and Error Logging

**As a** Product Team member,
**I want** EIA API failures and stale-data conditions to be logged with full context and surfaced in the health endpoint,
**so that** the ticker never silently shows outdated data and the team can diagnose feed issues within 60 seconds.

## Acceptance Criteria
- [ ] Every failed EIA API request generates a structured `ERROR` log entry containing: endpoint URL (with API key redacted), HTTP response code or error type, timestamp, and retry attempt number — satisfies **NFR-06**
- [ ] Log entries are emitted within 60 seconds of the failure and visible in the platform's monitoring output — satisfies **NFR-06**
- [ ] When all 3 retry attempts are exhausted and fallback to stale cache is triggered, a `WARN` entry explicitly states the cache age and lists the five cached regional price values being served — satisfies **NFR-04**, **NFR-05**
- [ ] **Security:** Log entries never include the raw `EIA_API_KEY` value; the key is fully masked or omitted from all log and diagnostic output — satisfies **NFR-07**
- [ ] An `/actuator/health` contributor (`EiaHealthIndicator`) reports `UP` when the cache is fresh (< 24h), `DEGRADED` when stale (24h–48h), and `DOWN` when data is absent or older than 48 hours — satisfies **US-EIA-05** (silent staleness detection)
- [ ] Cyclomatic complexity ≤ 10 in `EiaHealthIndicator.health()` — satisfies reviewer checklist quality gate
- [ ] JaCoCo branch coverage ≥ 80% including: UP (fresh cache); DEGRADED (stale 24h–48h); DOWN (no data or > 48h); key-masking confirmed in log output assertions
