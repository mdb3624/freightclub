# INF-103 — Platform-Wide Row Level Security (RLS) Infrastructure

**As a** Platform Administrator,
**I want** every data table on the Enon instance to have PostgreSQL Row Level Security enforced via the `app.current_tenant` session variable,
**so that** a misconfigured query, a missing WHERE clause, or a future developer mistake can never expose cross-tenant data — security is enforced at the database layer, not the application layer.

## Acceptance Criteria
- [ ] Every table (except public lookup tables) has a Flyway migration that executes `ALTER TABLE <name> ENABLE ROW LEVEL SECURITY` and `ALTER TABLE <name> FORCE ROW LEVEL SECURITY` — satisfies **REQ-103**
- [ ] A standard RLS policy `<table>_tenant_isolation` is applied to each table: `USING (tenant_id = current_setting('app.current_tenant', true)::uuid)` for SELECT, and `WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::uuid)` for INSERT/UPDATE
- [ ] Each persistence adapter (`JpaLoadAdapter`, `JpaCarrierAdapter`, etc.) executes `SELECT set_config('app.current_tenant', :tenantId, true)` as the first statement within every `@Transactional` block; a `finally` block is NOT required because `true` (transaction-local) ensures automatic revert on commit or rollback
- [ ] A `RlsPolicyTest` using Testcontainers (PostgreSQL 16) verifies that a direct JDBC query without `set_config` returns zero rows from any RLS-protected table; this test is mandatory CI gate — satisfies **NFR-502**
- [ ] Every new Flyway migration that creates a table must include the RLS boilerplate; a Checkstyle/PMD rule or ADR checklist item enforces this in code review
- [ ] Every RLS-protected table index leads with `tenant_id` as the first column, followed by `deleted_at` and the query-specific column — satisfies **NFR-503**
- [ ] Code must adhere to **NFR-501** (complexity ≤ 10): RLS setup logic must be encapsulated in a shared utility method with a single responsibility
