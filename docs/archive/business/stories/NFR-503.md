# NFR-503 — Database Scalability: Partial Indexes & Query Optimization

**As a** Platform Administrator,
**I want** all high-frequency query paths to use partial or composite indexes led by `tenant_id`,
**so that** query latency on the Enon PostgreSQL 16 instance remains below 100ms at the p99 percentile even as load volume scales to 100,000 rows per table.

## Acceptance Criteria
- [ ] Every table with a `status` column used in filtering has a partial index: `CREATE INDEX CONCURRENTLY idx_<table>_active ON <table>(tenant_id, status) WHERE deleted_at IS NULL` — satisfies **NFR-503**
- [ ] The load board query (`SELECT * FROM loads WHERE tenant_id = ? AND status = 'PUBLISHED' AND deleted_at IS NULL`) uses an index scan on the partial index; `EXPLAIN (ANALYZE, BUFFERS)` output stored as a test fixture for regression detection
- [ ] The `message_outbox` polling query uses a composite index on `(tenant_id, status, created_at)`; rows with `status = 'PROCESSED'` are excluded by the partial index `WHERE status IN ('PENDING', 'DEAD_LETTER')`
- [ ] All spatial indexes on `GEOGRAPHY` columns use `USING GIST`; non-spatial tenant-filtered indexes use `USING BTREE` — no default index type inference
- [ ] A `IndexCoverageTest` (Testcontainers) iterates all tables via `information_schema.tables` and asserts that no table with a `tenant_id` column is missing a `tenant_id`-leading index; this is a mandatory CI gate
- [ ] `VACUUM ANALYZE` is scheduled weekly on the Enon instance; autovacuum settings for `loads`, `message_outbox`, and `carrier_risk_profiles` are tuned: `autovacuum_vacuum_scale_factor = 0.01`
- [ ] Flyway migration naming convention ensures index migrations run after table migrations: `VYYYYMMDD_NNN__idx_<table>_<purpose>.sql`
- [ ] Code must adhere to **NFR-501** (complexity ≤ 10): any query-building helper that constructs dynamic SQL must not exceed complexity 10
