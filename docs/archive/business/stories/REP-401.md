# REP-401 — Mutual Post-Delivery Rating System

**As a** Shipper or Trucker,
**I want** to rate my counterpart after a load is delivered,
**so that** the platform builds a trust layer that helps truckers choose reliable shippers and shippers select high-performing carriers.

## Acceptance Criteria
- [ ] `ratings` table: `id UUID`, `tenant_id UUID`, `load_id UUID`, `reviewer_id UUID`, `reviewee_id UUID`, `reviewer_role VARCHAR`, `stars SMALLINT CHECK(1..5)`, `comment TEXT NULL`, `created_at TIMESTAMPTZ`; unique constraint on `(load_id, reviewer_id)` prevents double-rating; RLS on `tenant_id` — satisfies **REQ-103**; index on `(tenant_id, reviewee_id)` for aggregate queries — satisfies **NFR-503**
- [ ] Rating endpoints are unlocked only after the load reaches `DELIVERED` status; attempting to rate an in-progress load returns `409 Conflict` with error code `LOAD_NOT_DELIVERED`
- [ ] `POST /api/v1/ratings/{loadId}/trucker` requires `ROLE_SHIPPER`; `POST /api/v1/ratings/{loadId}/shipper` requires `ROLE_TRUCKER`; both persist a `ratings` row and write a `RatingSubmittedEvent` to `message_outbox` and `audit_log`
- [ ] `GET /api/v1/ratings/profile/{userId}` returns aggregate stats: `{ averageStars, totalRatings, completedLoads, recentComments[] }`; computed via a single SQL aggregate query, not in-memory — satisfies **NFR-503**
- [ ] Shipper reputation badge (average stars + avg payment speed label, e.g. "Typically pays in 7 days") is derived from `ratings` + `payments` tables and surfaced on load board cards so truckers can evaluate before claiming
- [ ] Cyclomatic complexity ≤ 10 in `RatingService.submitRating()` — satisfies **NFR-501**
- [ ] JaCoCo branch coverage ≥ 80% for: valid rating → stored; duplicate rating → 409; rating before delivery → 409 — satisfies **NFR-502**
