# DB-001 — Resize State Columns to CHAR(2) with CHECK Constraint

**As an** engineer,
**I want** `origin_state` and `destination_state` resized to `CHAR(2)` with a `CHECK` constraint,
**so that** full state names like "Illinois" cannot be stored and trucker state filters always return correct results.

## Acceptance Criteria

- [ ] A Flyway migration alters `loads.origin_state` and `loads.destination_state` from `VARCHAR(50)` to `CHAR(2)`.
- [ ] A `CHECK` constraint is added: value must match one of the 50 US state codes plus DC (e.g., `CHECK (origin_state REGEXP '^[A-Z]{2}$')`).
- [ ] Migration includes a data fix step that converts any existing full state names to their 2-letter codes before resizing the column.
- [ ] Any existing rows with unmappable state values are logged and flagged for manual review; migration does not silently delete data.
- [ ] Application-layer validation (LoadForm dropdown, S-005) is in place before this migration runs in production.
- [ ] Migration follows the project Flyway naming convention: `V{YYYYMMDD}_{seq}__{description}.sql`.
- [ ] After migration, inserting a load with `origin_state = 'Illinois'` fails at the database level.
