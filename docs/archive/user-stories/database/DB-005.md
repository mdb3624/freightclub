# DB-005 — CHECK Constraints for `loads` Enum Columns

**As an** engineer,
**I want** `CHECK` constraints on `loads.status`, `loads.equipment_type`, `loads.pay_rate_type`, and `loads.payment_terms`,
**so that** invalid values cannot be written via direct SQL or a future application bug that bypasses Java enum validation.

## Acceptance Criteria

- [ ] A Flyway migration adds `CHECK` constraints to all four columns.
- [ ] The allowed values in each constraint exactly match the application-layer Java enums (no drift).
- [ ] The migration includes a data audit step: any existing rows with values not in the allowed set are reported (migration fails loudly rather than silently corrupting or truncating data).
- [ ] After migration, a direct SQL `UPDATE loads SET status = 'INVALID_STATUS'` fails with a constraint violation.
- [ ] Constraint names follow a consistent pattern (e.g., `chk_loads_status`).
- [ ] Migration follows the Flyway naming convention.
