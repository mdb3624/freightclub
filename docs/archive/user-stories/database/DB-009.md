# DB-009 — Create `load_events` Table

**As an** engineer,
**I want** a `load_events` table designed and created now,
**so that** the status timeline feature (S-021) and Phase 2 notifications are built on a solid schema rather than a rushed ad-hoc design.

## Acceptance Criteria

- [ ] A Flyway migration creates `load_events` with at minimum: `id` (UUID PK), `tenant_id`, `load_id` (FK → loads), `event_type` ENUM (e.g., CREATED, PUBLISHED, CLAIMED, PICKED_UP, DELIVERED, CANCELLED, ISSUE_REPORTED), `actor_id` (FK → users, nullable for system events), `actor_role` ENUM (SHIPPER, TRUCKER, SYSTEM), `occurred_at` DATETIME NOT NULL, `metadata` JSON NULL (for extensible event data like cancellation reason, issue description).
- [ ] Index added on `(load_id, occurred_at)` for timeline queries.
- [ ] Index added on `(tenant_id, occurred_at)` for notification fan-out queries.
- [ ] On each load status transition, the application writes a corresponding `load_events` row.
- [ ] The table is append-only — events are never updated or deleted.
- [ ] Migration follows the Flyway naming convention.

## Enables
- S-021 (Load status timeline)
- S-020 / T-022 (Notifications)
