# DB-010 — Scope `users.email` Uniqueness to Tenant

**As an** engineer,
**I want** `users.email` uniqueness scoped to `(tenant_id, email)` instead of globally,
**so that** the same person can register as both a SHIPPER and a TRUCKER or join multiple tenants without being blocked by a global constraint.

## Acceptance Criteria

- [ ] A Flyway migration drops the existing global unique constraint on `users.email`.
- [ ] The migration adds a new composite unique constraint on `(tenant_id, email)`.
- [ ] Existing data is audited for conflicts within a tenant before the constraint is added; conflicts are reported (migration fails loudly if any exist).
- [ ] After migration, the same email can exist in two different tenants without conflict.
- [ ] After migration, the same email cannot appear twice within the same tenant.
- [ ] Application-layer registration logic is updated to check uniqueness within the tenant, not globally.
- [ ] Migration follows the Flyway naming convention.
