# DOC-502 — Proof of Delivery (POD) Photo Upload & Delivery Gate

**As a** Trucker (Owner-Operator),
**I want** to upload a Proof of Delivery photo at the destination,
**so that** the shipper has tamper-evident confirmation the freight was received, and the `DELIVERED` state transition is permitted only when a valid POD exists.

## Acceptance Criteria
- [ ] `POST /api/v1/documents/{loadId}/pod-photo` accepts `multipart/form-data`; validates MIME type (`image/jpeg`, `image/png`) and file size (≤ 25 MB); stores file via `StorageService` and writes a `load_documents` row with `document_type = 'POD'` — satisfies **REQ-203**
- [ ] `load_documents` table has `tenant_id` column with RLS policy; `SELECT` requires `tenant_id = app.current_tenant`; upload is restricted to `ROLE_TRUCKER` with a matching `carrierId` on the load aggregate — satisfies **REQ-103**
- [ ] Attempting `PUT /api/v2/loads/{id}/deliver` without a corresponding `load_documents` row where `document_type = 'POD'` and `load_id = :loadId` returns `422 Unprocessable Entity` with error code `POD_REQUIRED`
- [ ] A load may only have one active POD; a second upload for the same `load_id` replaces the prior record (sets `deleted_at = NOW()` on the old row) and writes a `DOCUMENT_REPLACED` event to `audit_log`
- [ ] Both the shipper and the trucker can retrieve the POD via `GET /api/v1/documents/{loadId}?type=POD` — requires `authenticated()` and matching `tenant_id`
- [ ] Offline-first: the frontend queues the POD upload request in IndexedDB when offline; service worker background-sync replays it on reconnect — satisfies **REQ-203**
- [ ] Code must adhere to **NFR-501** (complexity ≤ 10): document validation logic is extracted into a `DocumentValidationService` with a single responsibility
- [ ] JaCoCo branch coverage for upload and delivery-gate paths ≥ 80% — satisfies **NFR-502**
