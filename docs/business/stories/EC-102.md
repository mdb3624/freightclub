# EC-102 — Tenant Cross-Talk Prevention: RLS 404 Masking

**As a** Shipper or Trucker,
**I want** any request for a resource ID that belongs to a different tenant to return `404 Not Found` — not `401 Unauthorized` or `403 Forbidden`,
**so that** the existence of other tenants' data is never leaked through error code semantics.

## Acceptance Criteria
- [ ] When PostgreSQL RLS enforces `tenant_id = app.current_tenant`, a query for a cross-tenant row returns zero rows — identical to querying a non-existent row; the application layer must **not** distinguish between "row not found" and "row belongs to another tenant" — satisfies **REQ-103**
- [ ] `LoadRepository.findByIdAndTenantIdAndDeletedAtIsNull(id, tenantId)` returns `Optional.empty()` for both missing IDs and cross-tenant IDs; the service layer maps `Optional.empty()` to `LoadNotFoundException` → `404 Not Found` — never `403`
- [ ] The same 404-masking pattern is applied to all repository lookup methods across all modules: `carriers`, `documents`, `ratings`, `factoring_requests`, `audit_log`; a code review checklist gate enforces this
- [ ] `GlobalExceptionHandler` maps `LoadNotFoundException` (and all equivalent `EntityNotFoundException` subclasses) to `404`; it must **never** expose tenant-related context in the error body (no `tenantId` field in the JSON response)
- [ ] Integration test (MockMvc) verifies: authenticated SHIPPER queries a valid load ID from a different tenant → `404`; authenticated SHIPPER queries a genuinely missing ID → `404`; both return the same error body shape
- [ ] Cyclomatic complexity ≤ 10 in `GlobalExceptionHandler` — satisfies **NFR-501**
- [ ] Composite index on `(tenant_id, id, deleted_at)` ensures the 404 fast-path does not full-scan the table — satisfies **NFR-503**

## Related Stories
- INF-103 (RLS is the database enforcement layer for this behaviour)
- LOAD-201 (loads are the primary resource at risk of cross-tenant guessing)
- AUD-601 (suspicious cross-tenant probe patterns may warrant audit logging)
