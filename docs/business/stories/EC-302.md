# EC-302 — Appointment-Window Detention Timer Logic

**As a** Trucker (Owner-Operator),
**I want** the detention timer to start at the later of my actual arrival time or my scheduled appointment time,
**so that** I cannot be charged or claim detention pay simply because I arrived early for a future appointment window.

## Acceptance Criteria
- [ ] `loads` table includes `appointment_time TIMESTAMPTZ NULL`; if `NULL`, detention timer behaviour falls back to standard arrival-based logic (FIN-401); Flyway migration adds the column with a `NULL` default — satisfies **REQ-103** (column has `tenant_id` on the parent load)
- [ ] `DetentionTimerStart` Value Object (domain layer, no Spring/JPA) encapsulates the rule: `timerStart = MAX(arrivedAt, appointmentTime ?? arrivedAt)`; negative values (arrived after appointment) are valid and treated as `timerStart = arrivedAt`
- [ ] `FIN-401`'s `DetentionCalculationService` delegates timer-start resolution to `DetentionTimerStart` before computing the fee; the appointment window is injected at construction — satisfies the Strategy pattern required by the Reviewer Checklist
- [ ] Edge case: carrier arrives 3 hours before a 2-hour free-time window → detention clock does not start until appointment time + 2 hours; test asserts `detentionFee = 0` in this scenario
- [ ] Edge case: carrier arrives 30 minutes after appointment time → timer starts at `arrivedAt`; test asserts `detentionFee` is calculated from actual arrival — satisfies **NFR-502** (≥ 80% branch coverage)
- [ ] `DetentionTimerStart.resolve()` cyclomatic complexity ≤ 10 — satisfies **NFR-501**
- [ ] Shipper UI shows the effective timer-start time on the load detail page, not just the raw arrival time
