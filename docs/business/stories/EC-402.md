# EC-402 — Financial Event Idempotency via Outbox Request ID

**As the** Platform,
**I want** every financial event (invoice creation, payment disbursement, QuickPay advance, detention charge) to carry a unique `request_id` that prevents duplicate processing under poor-connectivity or retry conditions,
**so that** a carrier cannot be paid twice or charged twice for the same event due to a double-tap, network retry, or background-sync replay.

## Acceptance Criteria
- [ ] `message_outbox` gains a `request_id UUID NOT NULL` column with a `UNIQUE` constraint scoped per `(tenant_id, event_type, aggregate_id)`; Flyway migration adds the column and unique index — satisfies **REQ-103** and **NFR-503**
- [ ] All financial command handlers (`InvoiceCommandHandler`, `QuickPayCommandHandler`, `DetentionCommandHandler`) generate a deterministic `request_id` using UUID v5 (namespace + `tenantId + eventType + aggregateId + callerNonce`); the `callerNonce` is a client-supplied idempotency key from the `X-Idempotency-Key` HTTP header (optional; defaults to a server-generated UUID)
- [ ] When `OutboxRelayService` attempts to insert a row and the `(tenant_id, event_type, aggregate_id)` unique constraint is violated, the insertion is skipped (ON CONFLICT DO NOTHING); the relay logs a `DUPLICATE_EVENT_SUPPRESSED` warning to `audit_log` — satisfies **REQ-402**
- [ ] API endpoints that trigger financial events accept an optional `X-Idempotency-Key` header; if the same key is presented twice within 24 hours, the second response returns the cached result (HTTP 200 with `X-Idempotency-Replayed: true`) without re-executing the financial logic
- [ ] Cyclomatic complexity ≤ 10 in all command handlers — satisfies **NFR-501**; idempotency key resolution is extracted into `IdempotencyKeyResolver`
- [ ] `idempotency_cache` table: `request_id UUID PK`, `tenant_id UUID`, `response_body JSONB`, `expires_at TIMESTAMPTZ`; rows purged after 24 hours by a scheduled cleanup job; RLS enforces tenant isolation
- [ ] JaCoCo branch coverage ≥ 80% for: first call → processed; duplicate call → replayed; expired key → reprocessed — satisfies **NFR-502**
