# FIN-502 — QuickPay Factoring Logic & Margin Verification

**As a** Trucker (Owner-Operator),
**I want** to opt into QuickPay after delivery and receive same-day payment at a discounted rate,
**so that** I can manage cash flow without waiting 30–60 days for standard invoice settlement.

## Acceptance Criteria
- [ ] QuickPay is only available after load status reaches `DELIVERED` and a valid BOL document (`document_type = 'BOL'`) exists for the load
- [ ] `FactoringService` computes the net advance: `grossRate × (1 − factoringFee)` where `factoringFee` is configurable per tenant (default 3%); the domain `FactoringOffer` Value Object enforces that `netAdvance > 0`
- [ ] Margin verification checks that the ELROD profitability score (from MAT-302) remains positive after the factoring fee is deducted; if not, a `LowMarginWarning` is surfaced to the trucker before confirmation — not a hard block
- [ ] Trucker acceptance creates a `factoring_requests` row with `status = 'PENDING'` and `tenant_id`; RLS prevents cross-tenant reads
- [ ] Payment disbursement is simulated via an `OutboundPaymentEvent` written to `message_outbox`; integration with an actual payments rail is out of scope for this story
- [ ] All fee calculations are stored with full precision (`NUMERIC(10,4)`) in the DB — no floating-point columns
- [ ] `factoring_requests` table has a composite unique index on `(load_id, tenant_id)` to prevent duplicate QuickPay requests for the same load
