# INF-001: Wrap 20 Flyway Migrations in DO/IF Idempotency Blocks

## Problem Statement

Of 32 Flyway migrations in `db/migration/`, 20 lack idempotency guards. Running migrations in partial-failure states (network timeout, out-of-memory, constraint violation) leaves the database in an inconsistent state that blocks retry attempts.

**Risk:** Deployment failures become unrecoverable without manual database intervention. Redeployment fragile. Schema evolution stalls.

**Affected Migrations:** All V202604* and V202605* files that use raw `CREATE TABLE`, `CREATE INDEX`, `ALTER TABLE`, or `INSERT` statements without `IF NOT EXISTS` or conditional logic.

**AC-001-2:** Error handling prevents duplicate-key and already-exists errors  
```sql
DO $$ BEGIN
  CREATE TABLE my_table (...);

END $$;
```

**AC-001-3:** INSERT statements use upsert pattern or check row existence first  
```sql
DO $$ BEGIN
  IF NOT EXISTS (SELECT 1 FROM my_table WHERE id = 'x') THEN
    INSERT INTO my_table (id, ...) VALUES ('x', ...);
  END IF;
END $$;
```

## Implementation Notes

- **DO Block Pattern:** Wrap entire migration in `DO $$ ... EXCEPTION ... END $$;`
- **Exception Types:**
  - `duplicate_table` — table already exists
  - `duplicate_object` — index/constraint already exists
  - `undefined_object` — target table/column does not exist (for ALTER)
  - `unique_violation` — duplicate key on INSERT (use UPSERT or IF NOT EXISTS)
- **Rollback:** No explicit rollback needed (Flyway tracks version; re-running is idempotent)
- **Testing:** Use a test script to simulate partial failure (e.g., kill `psql` mid-transaction and retry)
