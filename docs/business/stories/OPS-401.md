# OPS-401 — Load Claiming & Atomic State Transition

**As a** Trucker (Owner-Operator),
**I want** to claim a published load with a single action,
**so that** the load is atomically reserved for me and no other carrier can claim it concurrently.

## Acceptance Criteria
- [ ] `PUT /api/v2/loads/{id}/claim` transitions status `PUBLISHED` → `CLAIMED`; attempting to claim a load in any other status returns `409 Conflict`
- [ ] The JPA adapter uses `SELECT FOR UPDATE` (`@Lock(LockModeType.PESSIMISTIC_WRITE)`) on the load row before the status check, preventing double-claim under concurrent requests
- [ ] `carrierId` is stored as a `CarrierId` Value Object on the aggregate; `null` carrierId at claim time throws `LoadDomainException`
- [ ] `tenant_id` on the load must match the claiming trucker's `tenant_id` (enforced by RLS + application-layer `findById(tenantId, loadId)` lookup)
- [ ] `LoadClaimedEvent` is drained and written to `message_outbox` within the same transaction
- [ ] Shipper receives an in-app notification within one relay cycle confirming which carrier claimed their load
- [ ] Load board (`/api/v1/board`) no longer returns the load after it is claimed
