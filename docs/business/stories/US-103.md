# US-103: Load CRUD (Create, Edit, Cancel, Publish)

**Phase:** 1 — Core Load Lifecycle  
**Status:** COMPLETED  
**Depends On:** US-101, US-102  
**Blocks:** US-104, US-201, all downstream phases

---

## User Story

As a **shipper**, I need to create, edit, publish, and cancel loads in the system so that I can post freight for truckers to claim and manage the full lifecycle of my shipments.

---

## Acceptance Criteria

| ID | Criterion | Status |
|---|---|---|
| AC1 | Shipper can create a draft load with origin, destination, cargo weight, equipment type, and rate | ✅ |
| AC2 | Shipper can edit a draft load before publishing | ✅ |
| AC3 | Shipper can publish a draft load to change status from DRAFT to POSTED | ✅ |
| AC4 | Published loads are visible on the load board to all truckers (subject to equipment matching) | ✅ |
| AC5 | Shipper can cancel a POSTED load if no trucker has claimed it yet | ✅ |
| AC6 | Load form validates required fields (origin, destination, weight, equipment, rate) | ✅ |
| AC7 | Load belongs to the shipper's tenant; other tenants cannot view or edit it | ✅ |
| AC8 | Cancelled loads are soft-deleted (marked with deleted_at timestamp) | ✅ |

---

## Business Rules

- Load statuses: DRAFT → POSTED → CLAIMED → IN_TRANSIT → DELIVERED (only valid transitions).
- A load cannot be edited after it is CLAIMED by a trucker.
- Origin and destination must be valid US addresses (format validation on coordinates).
- Weight must be > 0 and <= max trailer capacity (variant by equipment type).
- Rate (CPM: cents per mile) must be > 0.
- Equipment types: Flatbed, Van, Reefer, Tanker, etc. (restricted enum).
- Shipper cannot cancel a load once CLAIMED; trucker must cancel instead via US-104.
- Multi-tenant isolation: each shipper only sees and manages their own loads.

---

## Implementation Notes

- **Frontend:** `LoadForm.tsx` with Zod schema validation for all fields; `LoadsTable.tsx` displays shipper's loads.
- **Backend:** `LoadController` (POST /api/v1/loads, PUT /api/v1/loads/{id}, PATCH /api/v1/loads/{id}/publish, PATCH /api/v1/loads/{id}/cancel); `LoadApplicationService` orchestrates domain logic.
- **Database:** `loads` table with status enum, soft delete via `deleted_at`, tenant_id isolation via RLS.
- **Tests:** `LoadApplicationServiceTest.java` for CRUD operations, status transitions; frontend tests for form validation.

## Technical Acceptance

- ✅ Cyclomatic complexity < 10 (status transitions simple)
- ✅ No Lombok usage (standard Java POJOs)
- ✅ RLS enforced (shipper can only access own loads via tenant_id)
- ✅ Soft deletes: cancelled loads have `deleted_at = CURRENT_TIMESTAMP`
- ✅ All queries include `deleted_at IS NULL` filter
