# US-104: Load Board & Claiming Workflow

**Phase:** 1 — Core Load Lifecycle  
**Status:** COMPLETED  
**Depends On:** US-101, US-102, US-103  
**Blocks:** US-201, US-301, all downstream phases

## User Story

As a **trucker**, I need to browse published loads on the load board, filter by profitability and equipment match, and claim loads so that I can start earning by transporting freight.

## Business Rules

- Load board visibility: Only POSTED loads are visible; DRAFT, CLAIMED, IN_TRANSIT, DELIVERED are hidden.
- Equipment matching: Trucker can only see loads that match their equipment. A Reefer trucker sees all general loads + refrigerated loads. A Van trucker sees only general loads.
- Concurrent claiming: If two truckers claim the same load simultaneously, only one succeeds (pessimistic locking with `SELECT FOR UPDATE`).
- Claim transaction: Upon successful claim, load status changes to CLAIMED and a `ClaimEvent` is published (for notifications).
- Cancellation: Trucker can cancel within 1 hour of claiming; after 1 hour, shipper must manually cancel.
- Load board updates: Frontend polls `/api/v1/loads/available` every 30 seconds or uses SSE for real-time updates (placeholder for Phase 6).

## Implementation Notes

- **Frontend:** `LoadBoardTab.tsx` and `LoadBoardTable.tsx` render published loads; `useClaimLoad.ts` hook handles claim mutation with retry logic.
- **Backend:** `LoadController` (GET /api/v1/loads, GET /api/v1/loads/{id}, PATCH /api/v1/loads/{id}/claim, PATCH /api/v1/loads/{id}/cancel); `LoadApplicationService.claimLoad()` uses `@Lock(LockModeType.PESSIMISTIC_WRITE)` to prevent race conditions.
- **Database:** `loads` table with status enum; `load_claims` junction table (soft-deleted on cancellation).
- **Events:** `ClaimEvent` published on successful claim, consumed by `NotificationService` (Phase 2).
- **Tests:** `LoadApplicationServiceTest.concurrentClaimTest()` validates pessimistic locking; `LoadBoardE2E.spec.ts` (Playwright) tests claim workflow.
