# US-105: Load Status Transitions (Pick Up & Delivery)

**Phase:** 1 — Core Load Lifecycle  
**Status:** READY_FOR_DESIGN  
**Depends On:** US-103, US-104, US-305  
**Blocks:** US-401, US-501, Phase 7b (Financial Intelligence)

---

## User Story

As a **trucker**, I need to mark a load as "Picked Up" and "Delivered" so that the shipper is notified of progress and the load lifecycle advances toward completion and payment.

---

## Acceptance Criteria

| ID | Criterion | Status |
|---|---|---|
| AC1 | Trucker can mark a CLAIMED load as PICKED_UP, changing status to IN_TRANSIT. | ✅ |
| AC2 | Transition to PICKED_UP is BLOCKED unless a BOL_PHOTO exists for the load (enforces US-305 AC2). | ✅ |
| AC3 | Trucker can mark an IN_TRANSIT load as DELIVERED, changing status to DELIVERED. | ✅ |
| AC4 | Transition to DELIVERED is BLOCKED unless a POD_PHOTO exists for the load (enforces US-305 AC4). | ✅ |
| AC5 | The system records `picked_up_at` and `delivered_at` timestamps upon status transition. | ✅ |
| AC6 | Only the trucker assigned to the load can perform these transitions. | ✅ |
| AC7 | Shippers receive an in-app notification when a load status changes to PICKED_UP or DELIVERED. | ✅ |
| AC8 | All status transitions are recorded in the load's audit trail. | ✅ |

---

## Business Rules

- **Status Flow:** CLAIMED → (BOL Upload) → PICKED_UP (IN_TRANSIT) → (POD Upload) → DELIVERED.
- **Atomic Transitions:** Status and timestamp updates must be committed in a single transaction.
- **Validation:** Server-side validation MUST re-verify document existence even if UI has gates.
- **Notification Trigger:** Success transitions publish `LoadStatusChangedEvent` for Phase 2 consumption.

---

## Technical Context

- **API Endpoints:**
  - `PATCH /api/v1/loads/{id}/pickup`
  - `PATCH /api/v1/loads/{id}/deliver`
- **Data Model:**
  - `loads` table: Update `status`, set `picked_up_at` / `delivered_at`.
- **Security:**
  - Enforce `assignee_id == current_user_id`.
  - Enforce `tenant_id` isolation.
