# US-301: S3 File Storage & Signed Upload URLs

**Phase:** 3 — Document Management  
**Status:** COMPLETED  
**Depends On:** US-101  
**Blocks:** US-302, US-303, US-305

---

## User Story

As a **platform architect**, I need S3 signed upload URLs and managed file storage so that shippers and truckers can securely upload BOL and POD photos without the platform handling raw file bytes.

## Business Rules

- S3 signed upload URLs expire in 15 minutes (prevents reuse).
- Signed download URLs expire in 1 hour (prevents long-term file sharing).
- File size limit: 25 MB per file; enforced via S3 bucket policy.
- Allowed file types: JPEG, PNG, WebP (MIME type validation on upload).
- Tenant isolation: files for tenant A cannot be accessed by tenant B (enforced via IAM policies + S3 key structure).
- Cost optimization: lifecycle policy deletes unconfirmed uploads after 30 days.

---

## Implementation Notes

- **Backend:** `StorageService` interface with `LocalStorageService` (dev) and `S3StorageService` (prod) implementations; generates signed URLs using AWS SDK.
- **Frontend:** `useUploadBolPhoto`, `useUploadPodPhoto` hooks fetch signed URL, then upload directly to S3 via `axios.put()`.
- **AWS Setup:** S3 bucket with CORS policy, lifecycle rules, encryption at rest (SSE-S3), versioning disabled.
- **Monitoring:** CloudWatch logs track upload completion events; failed uploads are retried client-side.
- **Tests:** `StorageServiceTest.java` mocks AWS SDK; frontend tests verify signed URL flow.
