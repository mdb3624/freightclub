# US-302: Platform-Generated BOL

**Phase:** 3 — Document Management  
**Status:** COMPLETED  
**Depends On:** US-301  
**Blocks:** US-303, US-305

---

## User Story

As a **shipper**, I need the platform to auto-generate a standard Bill of Lading (BOL) based on load details so that I have a formal document to share with carriers without manually creating one.

---

## Acceptance Criteria

| ID | Criterion | Status |
|---|---|---|
| AC1 | Backend generates PDF BOL from load data (origin, destination, weight, cargo description) | ✅ |
| AC2 | BOL includes shipper company name, contact info, and standard legal disclaimers | ✅ |

| AC4 | Generated PDF is stored in S3 under the load's document path | ✅ |
| AC5 | Shipper can download the generated BOL from the load detail page | ✅ |

---

## Business Rules

- BOL is generated on-demand (not pre-generated at load creation).
- BOL template uses FreightClub standard format (logo, load number, date, shipper/carrier details).
- Generated BOL becomes the "source of truth" unless shipper uploads a custom BOL photo.
- Cache TTL: 5 minutes (NFR-504 compliant); cache invalidated on load edit.
- PDF library: iText or Apache PDFBox for server-side generation.
- BOL includes mandatory fields per ICC (International Commerce Commission) guidelines.

---

## Implementation Notes

- **Backend:** `BillingService.generateBol(loadId)` uses PDF generation library; stores in S3; caches result.
- **API Endpoint:** GET `/api/v1/loads/{id}/bol` returns PDF or signed S3 download URL.
- **Frontend:** Button "Download BOL" on load detail page calls endpoint and triggers browser download.
- **Caching:** `@Cacheable("bolCache", key = "#loadId", unless = "#result == null", 5 minutes)`.
- **Tests:** `BillingServiceTest.java` validates PDF generation; frontend tests verify download button.
