# User Story: US-401 — Bidirectional Rating System

**Phase:** 4  
**Status:** COMPLETED  
**Depends On:** US-103 (Load Board & Claiming Workflow)  
**Blocks:** US-402 (Shipper Reputation Profile & Aggregation)  

---

## Overview

Implement a bidirectional rating system allowing shippers to rate truckers and truckers to rate shippers after load completion. Ratings are immutable and support dispute resolution, with duplicate protection to prevent multiple ratings for the same load pair.

---

## Acceptance Criteria

**AC-401-1:** Trucker can rate shipper after load delivery
- **When:** Load status = DELIVERED
- **Then:** Trucker sees rating prompt (1-5 stars + optional comment)
- **Verify:** RatingController POST `/api/v1/ratings` accepts `(loadId, ratingValue, comment)`

**AC-401-2:** Shipper can rate trucker after load delivery
- **When:** Load status = DELIVERED  
- **Then:** Shipper sees rating prompt (1-5 stars + optional comment)
- **Verify:** RatingController POST accepts rating from shipper role

**AC-401-3:** Duplicate protection (one rating per load pair)
- **When:** User attempts to rate same load twice
- **Then:** Service raises `RatingAlreadyExistsException`
- **Verify:** Database constraint `UNIQUE(load_id, rater_id, load_party)` enforced

**AC-401-4:** Rating immutability
- **When:** Rating created
- **Then:** No UPDATE/DELETE allowed
- **Verify:** PostgreSQL RLS policy blocks mutations; schema has no soft-delete on ratings

**AC-401-5:** Multi-tenant isolation on rating queries
- **When:** Service queries ratings
- **Then:** Filter by `tenant_id = TenantContextHolder.getTenantId()`
- **Verify:** Integration test confirms cross-tenant ratings invisible

**AC-401-6:** Ratings include metadata (reviewer role, load details)
- **When:** Rating created
- **Then:** Store `reviewer_role` (TRUCKER, SHIPPER), timestamp, load context
- **Verify:** RatingResponse includes reviewer_role and created_at

**AC-401-7:** Timestamp all ratings (TIMESTAMPTZ)
- **When:** Rating created
- **Then:** created_at = CURRENT_TIMESTAMP (UTC)
- **Verify:** Database column type = TIMESTAMPTZ; tests confirm UTC timestamps

## Implementation Notes

### Database Schema (Flyway Migration)
```sql
CREATE TABLE load_ratings (
  id VARCHAR(36) PRIMARY KEY,
  load_id VARCHAR(36) NOT NULL UNIQUE,
  rater_id VARCHAR(36) NOT NULL,
  rater_role VARCHAR(20) NOT NULL, -- TRUCKER or SHIPPER
  rating_value INT NOT NULL CHECK (rating_value >= 1 AND rating_value <= 5),
  comment TEXT,
  tenant_id VARCHAR(36) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (load_id) REFERENCES loads(id),
  FOREIGN KEY (rater_id) REFERENCES users(id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id),
  UNIQUE(load_id, rater_id, rater_role)
);

-- Immutable: no DELETE allowed
CREATE POLICY load_ratings_no_delete ON load_ratings
  AS (restrictive) FOR DELETE TO freightclub_runtime
  USING (false);
```

### Domain Model
- `Rating` entity (domain)
- `RatingService` (application layer, RLS-aware)
- `RatingRepository` (persistence)
- Exception: `RatingAlreadyExistsException`, `RatingNotAllowedException`

### API Endpoints
- `POST /api/v1/ratings` — Create rating
- `GET /api/v1/ratings/{loadId}` — View rating for load
- `GET /api/v1/users/{userId}/ratings` — User's received ratings
