## User Story

**As a** *Trucker (owner/operator)*,  
**I want to** *securely configure my bank account details for ACH payouts*,  
**so that** *I can receive load settlement payments directly to my account.*

---

## Acceptance Criteria (Gherkin Style)

### AC-1: Trucker Can Add Bank Account

**Given** a logged-in trucker is on the Account Settings page  
**When** they click "Add Bank Account" and enter:
- Account holder name
- Routing number (9 digits)
- Account number (encrypted)
- Account type (Checking / Savings)
- Account nickname (optional, e.g., "Business Checking")

**Then** the system MUST:
- Validate routing number format (9 digits, valid ABA bank)
- Encrypt account number using tenant-specific key
- Create immutable `payment_accounts` record with:
  - `status = 'PENDING_VERIFICATION'`
  - `account_type` (Checking/Savings)
  - `last_four_digits` (masked, for display)
  - `is_primary = false` (not default until verified)
- Return confirmation: "Bank account added. Verification pending (1-2 business days)."
- **NOT** send any test micro-deposits yet

**Example:**
```
Trucker enters:
  Account holder: John Doe
  Routing: 021000021
  Account: ••••••••5678
  Type: Checking
  Nickname: "Main Payout Account"

System stores encrypted account_number, returns masked version.
```

---

### AC-2: Micro-Deposit Verification (Async)

**Given** a trucker has added a bank account (status = PENDING_VERIFICATION)  
**When** 24 hours have elapsed (background job runs)  
**Then** the system MUST:
- Initiate two micro-deposits ($0.01 and $0.02, total $0.03) to the bank account
- Create `payment_account_verifications` record tracking:
  - `verification_code` (unique ID to correlate deposit amounts)
  - `status = 'MICRO_DEPOSITS_SENT'`
  - `sent_at = CURRENT_TIMESTAMP`
  - `expected_arrival = CURRENT_TIMESTAMP + 2 business days`
- Send email to trucker: "Two small deposits ($0.01, $0.02) sent to your account. Confirm the amounts in the app within 7 days to activate payout."
- Update `payment_accounts.status = 'AWAITING_MICRO_DEPOSIT_CONFIRMATION'`

**Example:**
```
Background job at 2026-04-28 10:00 UTC:
  For each PENDING_VERIFICATION account created > 24 hours ago:
    Send deposits of $0.01 and $0.02
    Record in payment_account_verifications
    Email trucker confirmation
```

---

### AC-3: Trucker Confirms Micro-Deposit Amounts

**Given** a trucker has received two micro-deposits and is on Account Settings  
**When** they click "Verify Account" and enter:
- Deposit amount #1 (e.g., $0.01)
- Deposit amount #2 (e.g., $0.02)

**Then** the system MUST:
- Match entered amounts against `payment_account_verifications.micro_deposit_amounts`
- If both amounts match exactly:
  - Update `payment_accounts.status = 'VERIFIED'`
  - Update `payment_accounts.is_primary = true` (becomes default if first account)
  - Delete `payment_account_verifications` record (clean up)
  - Send email: "Bank account verified! You can now receive payouts."
- If amounts do NOT match:
  - Increment `verification_attempt_count`
  - If `attempt_count >= 3`:
    - Update status to 'VERIFICATION_FAILED'
    - Email: "Verification failed after 3 attempts. Please add account again."
  - Else return error: "Amounts do not match. Please try again (1 attempt remaining)."

**Example:**
```
Trucker enters: $0.01, $0.02
System checks: matches micro_deposit_amounts
Result: status = 'VERIFIED', is_primary = true
```

---

### AC-4: Trucker Can Manage Multiple Accounts

**Given** a trucker has one VERIFIED account (is_primary = true)  
**When** they add a second bank account  
**Then** the system MUST:
- Allow adding the second account (same flow as AC-1)
- Keep `is_primary = false` on the new account (old account remains primary)
- Display both accounts in "Manage Payment Methods" with:
  - Account holder, last four digits, type, status
  - "Set as Primary" button (only for VERIFIED accounts)
- Allow selecting which account receives next payout

**Example:**
```
Account 1 (Primary):     John Doe ••••5678 (Checking) VERIFIED ✓
Account 2 (Secondary):   John Doe ••••9012 (Savings)  AWAITING_MICRO_DEPOSIT_CONFIRMATION

Trucker can click "Set as Primary" on Account 2 (once verified).
```

---

### AC-5: Trucker Can Delete Unverified Account

**Given** a trucker has an account with status = 'PENDING_VERIFICATION' or 'VERIFICATION_FAILED'  
**When** they click "Delete Account"  
**Then** the system MUST:
- Set `deleted_at = CURRENT_TIMESTAMP` (soft delete)
- Remove from display in "Manage Payment Methods"
- NOT allow deletion of VERIFIED accounts (only soft-delete via archive flow)

**Example:**
```
Account status = VERIFICATION_FAILED
Trucker clicks "Delete"
Result: deleted_at set, no longer visible
```

---

### AC-6: Multi-Tenancy & Data Isolation

**Given** any trucker interacting with payment accounts  
**When** they query or modify payment data  
**Then** the system MUST:
- **ALWAYS** enforce `tenant_id = TenantContextHolder.current()`
- **NEVER** allow cross-tenant visibility of:
  - Routing numbers, account numbers
  - Verification codes
  - Micro-deposit amounts
- **Use RLS Policy:** `SELECT/INSERT/UPDATE WHERE tenant_id = app.current_tenant_id AND deleted_at IS NULL`
- **Use PostgreSQL row-level encryption** for sensitive fields (account_number, routing_number)

**Example:**
```
Trucker from Tenant A queries their accounts:
  SELECT * FROM payment_accounts WHERE tenant_id = ? AND deleted_at IS NULL
  
Trucker from Tenant B cannot see or modify Tenant A's accounts.
```

---

### AC-7: Compliance & Audit Trail

**Given** any payment account operation (add, verify, delete, set primary)  
**When** the operation completes  
**Then** the system MUST:
- Create immutable audit log entry in `payment_account_audit_log`:
  - `trucker_id`, `account_id`, `action` (ADD, VERIFY, DELETE, SET_PRIMARY)
  - `timestamp`, `ip_address`, `user_agent`
  - `status_before`, `status_after`
- Log sensitive operations (NOT the values, only the action):
  - "Account verification initiated"
  - "Account marked as primary"
  - "Account marked for deletion"
- **30-year retention:** Never delete audit logs

**Example:**
```
Audit Log:
  2026-04-28 10:15:30 | Trucker ABC123 | ADD_ACCOUNT | PENDING_VERIFICATION | IP: 192.168.1.1
  2026-04-28 10:16:00 | Trucker ABC123 | MICRO_DEPOSITS_SENT | AWAITING_VERIFICATION | IP: 192.168.1.1
  2026-04-28 10:20:00 | Trucker ABC123 | VERIFY_ACCOUNT | VERIFIED | IP: 192.168.1.1
```

## Technical Context

### Dependencies
- **Database:** PostgreSQL with RLS policies enabled
- **Encryption:** Tenant-specific AES-256 for sensitive account data
- **Payment Gateway:** Stripe (for micro-deposit initiation)
- **Background Jobs:** Scheduled task to send micro-deposits at 24-hour mark
- **Email Service:** Notification to truckers on status changes

### Non-Functional Requirements
- **Latency:** Account add/verify operations must complete within 500ms (excluding email send)
- **Availability:** Payment account service must be 99.9% available (tied to ACH processing SLA)
- **Security:** Account numbers encrypted at rest and in transit; no logging of unmasked values
- **Compliance:** SOC2 Type II, PCI DSS Level 1 (if storing card data in future)
