# User Story: US-708 — Direct Load Assignment to Carrier

**Phase:** 7  
**Status:** READY_FOR_DESIGN  
**Owner:** BA  
**Depends On:** US-707 (Shipper Preferred Carrier List)  
**Blocks:** None (terminal story)  
**Traceability:** REQ-7.2 (Direct Carrier Engagement)

---

## User Story

**As a** *Shipper*,  
**I want to** *directly assign a load to a specific preferred carrier*,  
**so that** *I can work with trusted partners for time-sensitive loads or negotiate loads outside the public load board.*

---

## Acceptance Criteria

**AC-708-1:** Shipper can assign load to preferred carrier
- **When:** Shipper creates a DRAFT or OPEN load and selects "Assign to Carrier"
- **Then:** Show list of preferred carriers (from US-707), select one, click "Assign"
- **Verify:** POST /api/v1/loads/{loadId}/assign-to-carrier accepts `(carrierId)`, creates assignment record

**AC-708-2:** Assigned load visibility restricted to chosen carrier
- **When:** Load assigned to carrier (load status remains OPEN but carrier_id is set)
- **Then:** Load hidden from public load board; visible ONLY to assigned carrier
- **Verify:** GET /api/v1/board excludes assigned loads; GET /api/v1/board/{loadId} returns 404 unless user is assigned carrier

**AC-708-3:** Assigned carrier receives notification
- **When:** Load assigned to carrier
- **Then:** Send email + in-app notification (US-201 event)
- **Verify:** NotificationService.notifyCarrierLoadAssigned() called with load details

**AC-708-4:** Assigned carrier can claim load (requires no competing)
- **When:** Assigned carrier views their assigned loads
- **Then:** Display dedicated "My Assigned Loads" section with "Claim" button
- **Verify:** GET /api/v1/carrier/assigned-loads returns loads where carrier_id = currentUser.id

**AC-708-5:** Shipper can revoke assignment (reassign to different carrier)
- **When:** Load still OPEN and shipper clicks "Reassign to Different Carrier"
- **Then:** Show carrier list, select new carrier, confirm "Reassign? (cancels previous assignment)"
- **Verify:** PUT /api/v1/loads/{loadId}/assign-to-carrier updates carrier_id, sends notification to old & new carrier

**AC-708-6:** Cache assignment visibility (5-minute TTL)
- **When:** GET /api/v1/carrier/assigned-loads called
- **Then:** Cache key = `assignedLoads:{carrierId}:{tenantId}`, TTL = 5m
- **Verify:** `@Cacheable` on query; `@CacheEvict` on assignment change

**AC-708-7:** Multi-tenant isolation
- **When:** Service queries assigned loads
- **Then:** Filter by `tenant_id = TenantContextHolder.getTenantId()`
- **Verify:** Carrier from Tenant A cannot see Tenant B's assigned loads

**AC-708-8:** Assigned load cannot be claimed by non-assigned carrier
- **When:** Non-assigned carrier tries to claim assigned load
- **Then:** Return 403 Forbidden (or load not visible)
- **Verify:** LoadService.claimLoad() checks `load.carrier_id == currentUser.id || load.carrier_id == NULL`

## Implementation Notes

### Database Schema (Flyway Migration)
```sql
-- Add column to existing loads table
ALTER TABLE loads ADD COLUMN assigned_carrier_id VARCHAR(36);
ALTER TABLE loads ADD CONSTRAINT fk_assigned_carrier 
  FOREIGN KEY (assigned_carrier_id) REFERENCES users(id);

CREATE INDEX idx_assigned_carrier ON loads(assigned_carrier_id, tenant_id, deleted_at)
  WHERE deleted_at IS NULL;
```

### Domain Model
- `Load` entity: Add `assignedCarrierId` field
- `LoadAssignmentService` (application layer, validates assignment, sends notifications)
- Load visibility logic: Filter by `assigned_carrier_id` when querying board

### API Endpoints
- `POST /api/v1/loads/{loadId}/assign-to-carrier` — Assign load to carrier
- `PUT /api/v1/loads/{loadId}/assign-to-carrier` — Reassign to different carrier
- `GET /api/v1/carrier/assigned-loads` — View loads assigned to me (paginated)
- `DELETE /api/v1/loads/{loadId}/assignment` — Shipper cancels assignment (reverts to OPEN for all)

### Load Board Filtering
- **Before:** GET /api/v1/board lists all OPEN loads
- **After:** GET /api/v1/board excludes loads where `assigned_carrier_id IS NOT NULL` (unless user is assigned carrier)

### Notification Flow
- **Event:** Load assigned to carrier
- **Handler:** Send email + in-app notification (US-201 pattern)
- **Content:** "You've been assigned load #{id}: {origin} → {destination}, ${rate}"

### Caching Strategy
- Cache name: `assignedLoads`
- TTL: 5 minutes
- Key: `{carrierId}:{tenantId}`
- Eviction: Triggered on assignment change (POST/PUT/DELETE)

## Edge Cases

1. **Shipper reassigns to different carrier** — Old carrier gets notification, new carrier gets notification
2. **Assigned carrier rejects/doesn't claim within X days** — Load reverts to OPEN (future feature: auto-timeout)
3. **Shipper deletes assigned load** — Notification sent to assigned carrier: "Load #{id} cancelled"
4. **Assigned carrier soft-deleted** — Assignment becomes null (load reverts to OPEN)
