# US-713: Shipper Company Profile Setup (Post-Registration)

**Phase:** Phase 7 (Carrier & User Management)
**Status:** READY_FOR_DESIGN
**Owner:** BA (Michael)
**Traceability:** REQ-1.1 (from REQUIREMENTS.md)
**Dependency:** US-101 (Multi-Tenant Registration)

---

## User Story

**As a** *Shipper (newly registered)*,
**I want to** *complete my company profile before posting loads*,
**so that** *truckers can assess my credibility and I can meet the minimum requirements to publish loads.*

---

## Business Rules

- BR-1: A shipper MUST complete their profile before any load can be published (not before DRAFT creation).
- BR-2: Required fields: company name, billing contact email, phone number, physical address (city, state, zip).
- BR-3: Optional fields: company logo, MC number, USDOT number, preferred payment terms.
- BR-4: Profile completeness % is displayed on the shipper dashboard.
- BR-5: Incomplete profiles may publish DRAFTs but the "Publish" action is gated until profile is ≥ 80% complete.
- BR-6: Profile is scoped to the tenant (one profile per tenant).

---

## Acceptance Criteria (Gherkin Style)

### AC-1: Shipper Sees Profile Completion Banner

**Given** a shipper logs in with an incomplete profile (< 80% complete)
**When** they land on the dashboard
**Then** the system MUST display a dismissible banner:
- "Your company profile is X% complete. Complete it to start posting loads."
- Banner links directly to the profile setup page.

---

### AC-2: Shipper Completes Required Profile Fields

**Given** a shipper is on the Profile Setup page
**When** they fill in:
- Company name (required, max 120 chars)
- Billing contact email (required, valid email format)
- Phone number (required, US format)
- City, State, ZIP (required)
- MC Number (optional, 6–8 digits)
- USDOT Number (optional, up to 8 digits)

**And** clicks "Save Profile"
**Then** the system MUST:
- Validate all required fields are present and correctly formatted
- Persist to `shipper_profiles` table with `tenant_id` (from TenantContextHolder), `created_at`, `updated_at`, `deleted_at = NULL`
- Recalculate and display profile completeness %
- Return confirmation: "Company profile saved."

**Example:**
```
Shipper fills in:
  Company: "Apex Freight Solutions LLC"
  Billing Email: billing@apexfreight.com
  Phone: (512) 555-0182
  City/State/ZIP: Austin, TX 78701

Profile completeness → 80% → load publishing unlocked.
```

---

### AC-3: Load Publish is Gated on Profile Completion

**Given** a shipper has a DRAFT load and their profile completeness is < 80%
**When** they click "Publish"
**Then** the system MUST:
- Block the publish action
- Return an error: "Complete your company profile (currently X% complete) before publishing loads."
- Provide a direct link to the profile setup page

---

### AC-4: Profile Completeness Calculation

| Field Completed         | Points |
|-------------------------|--------|
| Company name            | 20%    |
| Billing email           | 20%    |
| Phone number            | 15%    |
| Physical address        | 25%    |
| MC or USDOT number      | 15%    |
| Company logo (optional) | 5%     |

A shipper reaches the 80% gate without the optional logo or MC/USDOT — they are not required.

---

### AC-5: Multi-Tenancy Isolation

**Given** any shipper profile operation
**Then** the system MUST:
- Enforce `tenant_id = TenantContextHolder.current()` on all queries
- Use RLS: `WHERE tenant_id = app.current_tenant_id AND deleted_at IS NULL`
- Never expose one shipper's profile data to another tenant

## Technical Context

### Domain Entity
- **ShipperProfile** — `shipper_profiles` table
  - `id` VARCHAR(36), `tenant_id` VARCHAR(36), `company_name`, `billing_email`, `phone`, `city`, `state`, `zip`, `mc_number` (nullable), `usdot_number` (nullable), `logo_url` (nullable)
  - `completeness_pct` (computed or stored INTEGER)
  - `created_at`, `updated_at` TIMESTAMPTZ, `deleted_at` TIMESTAMPTZ (nullable)

### Dependencies
- US-101 (user must exist)
- US-103 (load publish gate references profile completeness)

### Non-Functional Requirements
- Profile save ≤ 500ms
- Completeness % recalculated on every save
- RLS enforced on all queries

---

## Testing Strategy (TDD)

### Unit Tests
- Completeness % calculation for all field combinations
- Validation: missing required fields, invalid email, invalid phone
- Publish gate: blocked when < 80%, allowed when ≥ 80%

### Integration Tests
- Profile created after registration → completeness = 0%
- Full profile save → completeness ≥ 80% → load publish unlocked
- Multi-tenant isolation (Tenant A cannot read Tenant B's profile)
