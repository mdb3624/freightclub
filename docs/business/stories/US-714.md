# US-714: Trucker Onboarding Checklist (Pre-Load-Claim Gate)

**Phase:** Phase 7 (Carrier & User Management)
**Status:** READY_FOR_DESIGN
**Owner:** BA (Michael)
**Traceability:** REQ-7.1 (from REQUIREMENTS.md)
**Dependency:** US-101 (Registration), US-701 (Carrier Profiles)

---

## User Story

**As a** *Trucker (newly registered)*,
**I want to** *complete a guided onboarding checklist*,
**so that** *I can be cleared to claim loads and shippers know I am a verified, ready carrier.*

---

## Business Rules

- BR-1: A trucker MUST complete the onboarding checklist before claiming any load from the board.
- BR-2: Checklist items: (1) Add at least one piece of equipment, (2) Add at least one preferred lane, (3) Set availability, (4) Confirm contact phone number.
- BR-3: Insurance certificate upload (US-721) is optional at onboarding but triggers a visible "Unverified" badge until uploaded.
- BR-4: Checklist progress is persisted and survives logout/login.
- BR-5: Truckers may browse the load board before completing onboarding but the "Claim" button is disabled with an inline message.
- BR-6: Once all required checklist items are complete, the trucker's status transitions from `ONBOARDING` to `ACTIVE`.

---

## Acceptance Criteria (Gherkin Style)

### AC-1: Trucker Sees Onboarding Checklist on First Login

**Given** a trucker has registered (US-101) and has not completed onboarding
**When** they first log in
**Then** the system MUST:
- Display an onboarding checklist modal or dedicated page
- Show progress: "2 of 4 steps complete"
- List required steps with completion checkmarks:
  - [ ] Add your equipment
  - [ ] Set preferred lanes
  - [ ] Set your availability
  - [ ] Confirm your phone number

---

### AC-2: Each Checklist Step Links to the Relevant Page

**Given** the trucker is viewing the onboarding checklist
**When** they click a step
**Then** they are navigated to the relevant feature:
- "Add equipment" → Profile → Equipment (US-701 AC-1)
- "Set preferred lanes" → Profile → Lanes (US-701 AC-2)
- "Set availability" → Profile → Availability (US-701 AC-3)
- "Confirm phone" → Profile → Contact Info

**And** when they return, the completed step is marked ✓ and progress updates.

---

### AC-3: Claim Button Disabled Until Onboarding Complete

**Given** a trucker with `status = ONBOARDING` is browsing the load board
**When** they view a load card
**Then** the "Claim" button MUST be:
- Rendered but disabled (not hidden)
- Display tooltip on hover: "Complete your onboarding checklist to claim loads."
- Show a banner at the top of the board: "Finish your setup to start claiming loads. X steps remaining."

---

### AC-4: Trucker Status Transitions to ACTIVE on Completion

**Given** a trucker completes all 4 required checklist steps
**When** the final step is saved
**Then** the system MUST:
- Set `trucker_status = 'ACTIVE'` on their user record
- Dismiss the onboarding banner
- Display a one-time success message: "You're all set! Start claiming loads."
- Enable the "Claim" button on the load board immediately (no page reload required)

---

### AC-5: Insurance-Unverified Badge

**Given** a trucker has achieved ACTIVE status but has not uploaded insurance (US-721)
**When** shippers view the trucker's public profile (US-701 AC-4)
**Then** the system MUST display:
- An amber badge: "Insurance Unverified"
- This does NOT block load claiming — it is informational only

---

### AC-6: Checklist Progress Persists Across Sessions

**Given** a trucker has completed 2 of 4 steps and logs out
**When** they log back in
**Then** the system MUST:
- Restore checklist state from the database (not in-memory/session)
- Show "2 of 4 steps complete" with the correct steps marked ✓

---

### AC-7: Multi-Tenancy Isolation

**Given** any trucker onboarding operation
**Then** the system MUST:
- Enforce `tenant_id = TenantContextHolder.current()` on all queries
- Use RLS: `WHERE tenant_id = app.current_tenant_id AND deleted_at IS NULL`

## Technical Context

### Domain Entity / Column Addition
- **User / TruckerProfile** — add `trucker_status` ENUM: `ONBOARDING`, `ACTIVE`, `SUSPENDED`
  - New truckers default to `ONBOARDING` on registration
  - Transition to `ACTIVE` when checklist complete

### Checklist Persistence
- `trucker_onboarding_checklist` table (or JSON column on trucker_profiles):
  - `trucker_id`, `equipment_added` BOOLEAN, `lane_added` BOOLEAN, `availability_set` BOOLEAN, `phone_confirmed` BOOLEAN
  - `completed_at` TIMESTAMPTZ (nullable — set when all steps done)

### Dependencies
- US-101 (user exists, role = TRUCKER)
- US-701 (equipment, lane, availability steps delegate to carrier profile flows)
- US-721 (insurance upload — optional gate, badge only)

### Non-Functional Requirements
- Checklist state load ≤ 200ms on login
- Status transition atomic (single DB write)
- Claim-gate check enforced server-side (not just UI)

---

## Testing Strategy (TDD)

### Unit Tests
- Checklist completion logic (all 4 steps → ACTIVE, any missing → ONBOARDING)
- Claim blocked when status = ONBOARDING
- Claim allowed when status = ACTIVE

### Integration Tests
- Full registration → checklist steps → ACTIVE transition → claim succeeds
- Partial checklist + logout + login → state restored correctly
- Claim attempt with ONBOARDING status → 403 with correct error message
- Multi-tenant isolation
