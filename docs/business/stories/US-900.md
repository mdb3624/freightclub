# US-900: E2E Testing Infrastructure & Standards

**Epic:** Infrastructure & Testing Standards  
**Status:** DESIGN_APPROVED  
**Phase:** Cross-Phase (All)  
**Depends On:** None (foundational)  
**Deadline:** 2026-06-07 (Weeks 1-3)  
**Owner:** QA & Engineering Team  

---

## Business Context

### Problem Statement
Flaky integration tests reduce CI/CD reliability (currently 6/10 passes in CI). Manual testing succeeds but automated tests fail randomly due to:
- Race conditions (parallel tests sharing DB state)
- Hard-coded waits (brittle timing assumptions)
- UI-driven test setup (slow, error-prone)
- Fragile CSS/XPath selectors (break on style changes)
- No debugging visibility (no trace files for failure analysis)

### Impact
- Developers waste time debugging phantom test failures
- CI/CD blockers delay feature deployment
- QA can't pinpoint root causes (manual investigation)
- Test maintenance burden grows as codebase scales

### Desired Outcome
- Reliable tests: 10/10 CI passes (currently 6/10)
- Fast execution: 2-5s per test (currently 30s+ with UI setup)
- Easy debugging: Trace files with network/DOM analysis
- Resilient selectors: data-testid survives CSS refactors

---

## Business Rules

**BR-1:** All interactive UI elements must have `data-testid` attributes for selector resilience.

**BR-2:** Test data setup must use API calls (not UI automation) to avoid timeout issues.

**BR-3:** Test assertions must use web-first patterns (explicit waits, not hard-coded timeouts).

**BR-4:** All test failures must generate trace files (Playwright Inspector format) for debugging.

**BR-5:** Multi-tenancy must be explicitly tested (no cross-tenant data leakage).

**BR-6:** Test infrastructure must be documented (DEBUGGING_GUIDE, fixture API, component specs).

## Acceptance Criteria

### AC-1: Component Data-testid Coverage
**Given** all interactive components in the codebase  
**When** a PR introduces new interactive elements (input, button, select, etc.)  
**Then** those elements MUST have `data-testid` attributes following naming convention  
**And** grep verification confirms 100% coverage  
**Verified By:** CODER Phase 1 + REVIEWER audit

### AC-2: Backend Test Endpoints Functional
**Given** E2E tests need to seed test data  
**When** global-setup.ts or tests call `/api/test/auth/register`, `/api/test/users/{id}`  
**Then** endpoints exist, return valid responses, and set required headers (Set-Cookie)  
**And** endpoints are gated with `@Profile("test")` to prevent production exposure  
**Verified By:** CODER Phase 2 + curl/Postman testing

### AC-3: Refactored Tests Pass Locally
**Given** the refactored login test file (7 test cases)  
**When** `npm run test:e2e -- login-integration-refactored.spec.ts` runs  
**Then** all 7 tests pass (green status)  
**And** trace files are generated in `test-results/trace-*.zip`  
**And** no hard-coded waits are present  
**Verified By:** CODER Phase 3 execution

### AC-4: Web-First Assertions Mandatory
**Given** all E2E tests in the suite  
**When** tests check for element visibility, state, or content  
**Then** assertions use web-first patterns: `expect().toBeVisible()`, `expect().toBeEnabled()`  
**And** NO hard-coded `waitForTimeout()` calls exist  
**And** assertion timeouts are explicit (3-5s for UI, 5-10s for API responses)  
**Verified By:** CODER Phases 3-4 + REVIEWER hard gate

### AC-5: Trace Files Generated on Failure
**Given** any E2E test fails  
**When** test execution completes  
**Then** a Playwright trace file is saved to `test-results/trace-{test-name}-{timestamp}.zip`  
**And** trace contains: network requests, DOM snapshots, console logs, screenshots  
**And** DEBUGGING_GUIDE.md provides analysis workflow  
**Verified By:** CODER Phase 3 + manual test failure

### AC-6: Debugging Workflow Enables Root Cause Analysis
**Given** a test failure with trace file  
**When** team follows DEBUGGING_GUIDE.md workflow  
**Then** team can identify root cause without manual reproduction  
**And** trace analysis shows: API status code, response payload, DOM state at failure  
**And** backend logs can be correlated to request timestamp  
**Verified By:** CODER Phase 3 + DEBUGGING_GUIDE workflow validation

### AC-7: CI/CD Integration Passes
**Given** GitHub Actions workflow includes E2E tests  
**When** tests run in CI environment  
**Then** pass rate is 10/10 (100%)  
**And** test artifacts (trace files) are uploaded on failure  
**And** Playwright HTML report is generated  
**Verified By:** CODER Phase 6 + CI execution

## Implementation Phases

### Phase 1: Component Updates (Week 1, 2 hours)
Add `data-testid` attributes to all interactive components (AC-1)

**Deliverables:**
- LoginPage.tsx: email-input, password-input, login-submit-btn, error messages
- Dashboard container: dashboard-container
- All interactive form elements: {field}-input, {action}-btn, {page}-container

### Phase 2: Backend Verification (Week 1, 1 hour)
Verify test endpoints exist and are functional (AC-2)

**Deliverables:**
- POST `/api/test/auth/register` returns userId + tenantId
- DELETE `/api/test/users/{id}` accessible for cleanup
- Endpoints gated with `@Profile("test")`

### Phase 3: Test Execution (Week 1, 30 min)
Run refactored tests locally (AC-3)

**Deliverables:**
- All 7 tests pass: `npm run test:e2e -- login-integration-refactored.spec.ts`
- Trace files generated in test-results/

### Phase 4: Original Test Replacement (Week 1, 15 min)
Replace original tests with refactored versions (AC-4)

**Deliverables:**
- Original `login-integration.spec.ts` replaced
- Full test suite passes: `npm run test:e2e`

### Phase 5: Pattern Rollout (Week 2, ongoing)
Apply refactored pattern to other E2E tests (AC-4, AC-5, AC-6)

**Deliverables:**
- All E2E tests migrated to: data-testid + web-first + API fixtures + traces

### Phase 6: CI/CD Integration (Week 3, 1 week)
Integrate tests into GitHub Actions (AC-7)

**Deliverables:**
- GitHub Actions workflow runs E2E tests
- Trace artifacts uploaded on failure
- Playwright HTML report generated
- All tests pass in CI (10/10)
