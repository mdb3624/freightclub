# CHG-001: US-707 Preferred Carriers — Missing Nav Header & Non-Functional Carrier Search

**Original Story:** US-707 (Shipper Preferred Carrier List)
**Discovered By:** CODER on 2026-06-04
**Root Cause:** Two gaps shipped through E2E sign-off because tests verified API interactions only, not full UI rendering or functional search behavior.

---

## Issue 1: Missing Navigation Header

**Technical Blocker:** `PreferredCarriersList` is not wrapped in `AppShell`, so the page renders with no nav bar, user name, or sign-out button — inconsistent with every other authenticated page in the app.

**Root Cause:** The route in `App.tsx` wraps `PreferredCarriersList` directly in `ProtectedRoute` without `AppShell`. All other authenticated pages (ShipperDashboard, ProfilePage, etc.) use `AppShell` internally. This one was missed.

**Fix:** Import and wrap `PreferredCarriersList` return value with `<AppShell>`.

---

## Issue 2: Carrier Search Uses Hardcoded Mock Data

**Technical Blocker:** `AddCarrierModal.handleSearch()` filters a static array of 3 fake carriers (`FedEx Freight`, `XPO Logistics`, `J.B. Hunt`). Real carriers registered in the system are completely unsearchable. The "Add Carrier" flow is non-functional for production use.

**Root Cause:** No backend endpoint exists for carrier search. The modal was built with a placeholder comment: *"In a real app, this would call an API endpoint to search carriers"* — this was never implemented before story sign-off.

**Fix Required:**
1. Add `findByRoleAndDeletedAtIsNull(UserRole role, Pageable pageable)` + search query to `UserRepository`
2. Add `GET /api/v1/shippers/carriers/search?q={term}` backend endpoint (searches TRUCKER users by first name, last name, or email within tenant)
3. Replace mock data in `AddCarrierModal.handleSearch()` with real API call
