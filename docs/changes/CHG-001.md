# CHG-001: US-707 Preferred Carriers ŌĆö Missing Nav, Non-Functional Search, Modal Redesign Required

**Original Story:** US-707 (Shipper Preferred Carrier List)
**Discovered By:** CODER on 2026-06-04
**Root Cause:** Three gaps shipped through E2E sign-off because tests verified API interactions only, not full UI rendering, functional search behavior, or UX usability.

---

## Issue 1: Missing Navigation Header

**Technical Blocker:** `PreferredCarriersList` is not wrapped in `AppShell`, so the page renders with no nav bar, user name, or sign-out button ŌĆö inconsistent with every other authenticated page in the app.

**Root Cause:** The route in `App.tsx` wraps `PreferredCarriersList` directly in `ProtectedRoute` without `AppShell`. All other authenticated pages use `AppShell` internally. This one was missed.

**Fix:** Wrap `PreferredCarriersList` return value with `<AppShell>`.

---

## Issue 2: Carrier Search Uses Hardcoded Mock Data

**Technical Blocker:** `AddCarrierModal.handleSearch()` filters a static array of 3 fake carriers (`FedEx Freight`, `XPO Logistics`, `J.B. Hunt`). Real carriers registered in the system are completely unsearchable. The "Add Carrier" flow is non-functional for production use.

**Root Cause:** No backend endpoint exists for carrier search. The modal was built with a placeholder comment: *"In a real app, this would call an API endpoint"* ŌĆö never implemented before sign-off.

**Fix Required:**
1. Add search query to `UserRepository` (search TRUCKER users by name/email within tenant)
2. Add `GET /api/v1/shippers/carriers/search?q={term}` backend endpoint
3. Replace mock data in `AddCarrierModal.handleSearch()` with real API call

---

## Issue 3: Add Carrier Modal UX Redesign Required

**UX Blocker:** The current modal provides only a text input that does nothing (no real search, no results list, no way to identify and select a carrier). Even if the backend search endpoint is added, the modal flow needs a complete redesign:

- A search input that triggers live results as you type
- A results list showing carrier name, email, equipment type
- A clear "select" action per result before submitting
- Feedback when no carriers match
- Empty state / loading state

**Root Cause:** The modal was designed around mock data with no thought for how a shipper actually finds a carrier they want to add. The UX was never reviewed by HFD before implementation.

**Fix:** HFD must design the Add Carrier modal interaction flow before CODER implements the real search. This is a **Sequential Lock gate violation** ŌĆö CODER implemented without HFD sign-off on this interaction.
