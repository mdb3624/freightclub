## CHG-849: Duplicate Carrier Dashboard Epic (US-730 built twice)

**Original Story:** US-730 (Carrier Dashboard MVP epic)
**Discovered By:** CODER, during OO/carrier workflow investigation, 2026-07-04
**Root Cause:** The US-730-0 epic was designed and built as `CarrierDashboard.tsx` (a locked-HFD-spec-compliant but 100%-mock component, routed at `/dashboard/carrier`) without checking that an equivalent, real, hook-driven dashboard (`TruckerDashboard.tsx`, routed at `/dashboard/trucker`) already existed and was live. Nothing in the app navigates to `/dashboard/carrier` — it is orphaned.
**Technical Blocker:** None — this is a documentation/planning defect, not an implementation blocker. `Story_Map.md` listed the entire epic as `READY_FOR_DESIGN`, which was stale and caused a "ground-up rebuild" plan to be proposed before the real implementation was discovered.

**Options:**
1. Retire `CarrierDashboard.tsx` + `/dashboard/carrier` route, correct `Story_Map.md` to reflect `TruckerDashboard.tsx` as the real implementation, and close the epic as substantially complete (only Payment Acknowledgment remained genuinely open, and its scope narrowed once the existing Stripe invoice system was found).
2. Keep both dashboards and reconcile design differences between them.

**Recommendation:** Option 1 — Option 2 has no benefit; `TruckerDashboard.tsx` is real and already serves users, `CarrierDashboard.tsx` never has.

**Next Steps:**
1. CODER diffs `CarrierDashboard.tsx` against `TruckerDashboard.tsx` for any visual gap (Task 6 of the 2026-07-04 implementation plan).
2. CODER removes the orphaned route/component if no unique design is found.
3. LIBRARIAN corrects `Story_Map.md` (done in this ticket, Task 1).

**Status:** CHG-849 RESOLVED — Option 1 approved by user 2026-07-04 (see `docs/superpowers/specs/2026-07-04-carrier-oo-workflow-verification-design.md`).
