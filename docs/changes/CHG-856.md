## CHG-856: Document Storage Is Broken in Production — Ephemeral `/tmp` on Cloud Run + Unpersisted Content-Type

**Original Story:** US-302 (Platform-Generated BOL), US-303 (BOL/POD Photo Upload & Viewing) — both COMPLETED, both affected
**Discovered By:** CODER, during live browser + API verification of the CHG-855 BOL workflow, 2026-07-14
**Root Cause:** Two independent, confirmed-live bugs, verified by creating a real load via the API against the Docker test environment and inspecting the actual file on disk and the actual HTTP response:

1. **Every document download 400s, in every environment.** `LoadDocument.contentType` (`backend/src/main/java/com/freightclub/domain/LoadDocument.java:39-40`) is annotated `@Transient` — it is set in memory at upload/generation time but never persisted to the `load_documents` table. On any subsequent read (i.e. always, since JPA reloads the entity), `getContentType()` returns `null`. `DocumentController.downloadDocument()` (`DocumentController.java:63-71`) calls `MediaType.parseMediaType(content.contentType())`, which throws on `null`. **Live reproduction:** created a load via `POST /api/v1/loads`, confirmed a genuine 1398-byte `%PDF-1.6` file was written to disk, then called `GET /api/v1/documents/file/{id}` and got `400 {"message":"Invalid mime type \"null\": 'mimeType' must not be empty"}`. This affects `BOL_GENERATED`, `BOL_PHOTO`, `POD_PHOTO`, and `ISSUE_PHOTO` — not a BOL-specific bug.

2. **Production storage path is ephemeral and non-shared — confirmed misconfigured, not just theoretically wrong.** `LocalStorageService` (`backend/src/main/java/com/freightclub/storage/LocalStorageService.java`) writes to local disk at `${app.storage.local-path:${user.home}/freightclub-uploads}`. `application-prod.yml:27-28` explicitly sets `app.storage.local-path: /tmp/freightclub-uploads`. Per your `.claude/skills/start/SKILL.md`, real production is Google Cloud Run (no local volumes) — Cloud Run's `/tmp` is an in-memory tmpfs that is wiped on every instance restart, redeploy, or scale-to-zero event, and is never shared across concurrent instances. Any uploaded/generated document in production today is at risk of silently disappearing the moment the serving instance recycles, and is unreachable from any other instance handling a later request for the same document. Separately, `docker-compose.prod.yml:33` sets a `STORAGE_PATH` env var that is never actually consumed (Spring binds `app.storage.local-path`, not `STORAGE_PATH` — same class of env-var/property-name mismatch as the EIA wiring bug, FREIG-116/US-854) — and that compose file isn't even how production is deployed, so it's dead code either way.

**Technical Blocker:** This isn't a BOL-specific gap — it blocks the entire document-attachment feature set (BOL, BOL photo, POD photo, issue report photos) in production, and blocks CHG-855's pickup-attestation feature from being verifiable in prod even after that work ships, since the underlying storage/retrieval layer it depends on is broken there today.

**Options:**
1. **Storage abstraction + Google Cloud Storage backend for prod (recommended).** Extract a `StorageService` interface from `LocalStorageService`'s existing two methods; keep `LocalStorageService` unchanged and active via `@Profile({"dev","test"})` (this already works correctly on Windows and in the Docker test containers — no behavior change there); add a new `GcsStorageService implements StorageService` active via `@Profile("prod")`, using the `google-cloud-storage` Java client authenticating via Cloud Run's attached service account (Application Default Credentials — no key file to manage). Fix `LoadDocument.contentType` to a real persisted column as part of the same work, since both storage backends need it to serve downloads correctly.
2. **Mount a persistent volume to Cloud Run** (Cloud Storage FUSE or Filestore/NFS via Cloud Run gen2 volume mounts). Rejected: higher operational complexity (FUSE performance caveats, Filestore cost/VPC requirements) for no benefit over a direct GCS SDK integration, which is simpler and is the standard pattern for this kind of workload.
3. **Do nothing.** Rejected: BOL/POD documents carry legal/evidentiary weight (see CHG-855); silently losing them in production is not an acceptable risk to carry forward, and it directly undermines the attestation-lock feature CHG-855 is building.

**Recommendation:** Option 1.
