## CHG-857: Stale SPA-Shell Caching Broke In-App Navigation in Production

**Original Story:** US-822 (Shipment Status Panel), US-824 (Quick Actions Panel) — both COMPLETED, both affected
**Discovered By:** User report on production (`mdbfreightclub.com`) — "My Documents" and per-load selection both silently landed on the home page instead of the intended route, 2026-07-16
**Root Cause:** Two independent, confirmed-live bugs:

1. **Stale cached SPA shell + content-hashed asset chunks.** `index.html` was served with no `Cache-Control` header, so browsers could cache it. It references content-hashed Vite build asset filenames that are deleted on every redeploy. Once a redeploy landed, a browser holding a cached pre-deploy `index.html` would attempt a client-side lazy-route navigation (`React.lazy()` dynamic `import()`), which 404'd against the now-deleted chunk for that route — silently failing the navigation. Full page loads (which always re-fetch `index.html`) were unaffected; only in-app link/button clicks broke, which is why the bug looked selective and confusing to reproduce.
2. **Missing `/shipper/payments` route.** The dashboard's "Payments" quick action linked to `/shipper/payments`, which was never registered in `App.tsx` — it hit the catch-all route and silently redirected home. Independent of bug #1, found during the same investigation.

An initial investigation theory (stale Service Worker inherited from a prior site on the domain) was pursued and disproven via direct evidence (fresh browser session with zero history reproduced the bug; the "Hauler" content in question was confirmed to be FreightClub's own `TruckerLandingPage.tsx`, not an external site) before the real root cause was found via a dispatched subagent.

**Technical Blocker:** Not story-specific — this breaks in-app navigation platform-wide after every deploy, for any browser that cached `index.html` before the deploy landed. Directly undermined user trust in the shipper dashboard (US-822/US-824's core interactions).

**Also discovered mid-fix:** the corrected E2E spec's real bug — `frontend/e2e/shipper-documents-routing.spec.ts` (added with the original US-822/US-824 navigation work, PR #39) used `page.goto()` directly to destination URLs instead of clicking through the dashboard UI, so it never exercised the `onClick` → `navigate()` path where this regression actually lived, and passed the entire time the bug was live in production.

**Options:**
1. **Fix `Cache-Control` headers on the SPA shell + backfill the missing route (recommended, implemented).**
2. Add a build-time cache-busting query param / service-worker-based update-on-reload mechanism. Rejected as unnecessary — a correct `no-cache, must-revalidate` header on `index.html` alone is the standard, minimal fix; the browser's own conditional-request (ETag/Last-Modified) handling takes over from there.
3. Do nothing. Rejected: silent navigation failures directly following every deploy is an unacceptable, recurring UX regression.

**Recommendation:** Option 1.

**Status:** CHG-857 RESOLVED — deployed to production and live-verified 2026-07-16.

- Root cause fix applied in `frontend/docker-entrypoint.sh` — the file that actually generates the running nginx config at container start (`frontend/nginx.conf` is a template copied into the image but never read at runtime; kept in sync for documentation). Both the `/` SPA-fallback location and the exact-match `/index.html` location now set `Cache-Control: no-cache, must-revalidate`.
- Added `frontend/src/pages/PaymentsPlaceholder.tsx` (following the existing `QuoteRequestPlaceholder` pattern) and registered `/shipper/payments` in `App.tsx`.
- Rewrote `shipper-documents-routing.spec.ts` to click through the real dashboard UI (shipment row → View Documents, quick action → My Documents) instead of `page.goto()`-ing straight to the destination — verified 2/2 passing locally against the Docker test environment. Documented the new mandatory rule (navigation-verifying E2E specs must click through the UI, not `goto()` to the destination) in `.claude/rules/testing_standards.md`.
- **Deploy-blocking issue found and fixed during rollout:** the first deploy attempt failed — `docker-entrypoint.sh` had CRLF line endings (no `.gitattributes` existed to prevent a Windows checkout/edit from introducing them), which broke the container's `#!/bin/sh` shebang (`failed to load /docker-entrypoint.sh: no such file or directory` is the exact symptom of a CRLF shebang). Added `.gitattributes` pinning `.sh` files to LF, fixed the file, rebuilt, redeployed clean. Cloud Run's rollout safety meant the failed intermediate revision never received traffic — no user-facing impact from this deploy hiccup.
- **Live production verification (not mocked), per `testing_standards.md`'s external-config-wiring gate:** logged in as a real shipper user (`shipper@test.com`) against `https://mdbfreightclub.com`, confirmed via real browser automation:
  1. "My Documents" quick action → lands on `/shipper/documents` with real document data (previously redirected home).
  2. "View load" link from the documents list → lands on `/shipper/loads/:id` (Load Detail page) with real load data (previously redirected home).
  3. "Payments" quick action → lands on `/shipper/payments` showing the new placeholder (previously hit the catch-all/home page).
- Deployed `freightclub-frontend` revision `freightclub-frontend-00048-qc9` (image digest `sha256:a46090f8ec05fb7a07964fcd58ff90435be290e3e34715a17827e78700acdf55`) to Cloud Run, serving 100% of traffic.
