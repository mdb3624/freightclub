## CHG-859: `/api/v1/loads` vs `/api/v2/loads` Dual-Controller Coexistence — Decision

**Original Story:** None — discovered via `docs/project/PROJECT_AUDIT_2026-07-23.md` (independent repo audit), not mid-implementation of a specific story. Supersedes/resolves the standing `.claude/learnings.md` Technical Debt Ledger row: *"Dual `LoadController` coexistence... RLS and event propagation only apply to v2 endpoints"* (that row predates this decision and is now partially stale — see Root Cause).

**Discovered By:** Repo-wide audit + code review, 2026-07-23. Backend review flagged `/api/v2/loads` (`com.freightclub.modules.load.infrastructure.rest.LoadController`) as having zero authorization (no `SecurityConfig` matcher, no ownership check) — fixed separately as PROJECT_AUDIT item 1 (branch `fix/v2-loads-authorization`). This ticket addresses the *architectural* question that fix didn't resolve: which controller is the one going forward.

**Root Cause:** The codebase is mid-migration from a flat `controller/service/repository` package layout to a modular hexagonal structure (`modules/{carrier,load,payment,shipper}/{domain,application,infrastructure,presentation}`). `modules/load`'s domain/application layers are actively developed (9+ recent commits, full test coverage: `LoadAggregateTest`, `CreateLoadCommandTest`, `LoadAssignmentServiceTest`, etc.), but its REST adapter (`infrastructure/rest/LoadController`, mounted at `/api/v2/loads`) was not: only 2 commits total, last touched 2026-05-06 (2.5+ months stale relative to this audit), had no controller-level test until this audit added one, and — confirmed via grep — **zero references from `frontend/src` anywhere**. Meanwhile `/api/v1/loads` (`com.freightclub.controller.LoadController`, backed by `LoadService`) is the live, tested, frontend-wired path.

The existing ledger note that "RLS and event propagation only apply to v2 endpoints" is now stale: US-857 made RLS enforcement a property of the `freightclub_runtime` DB role (`BYPASSRLS` revoked, real tenant policies added) — it applies to *all* queries through that role regardless of which controller issued them, not something v1 lacks. What v2 genuinely has that v1 doesn't is domain event publishing (`DomainEventPublisher`) and a cleaner aggregate/use-case split — real architectural value, just not yet load-bearing for actual traffic.

**Technical Blocker:** None — this is not blocking any in-flight story. It's an architecture decision that was undocumented (two controllers silently coexisting with divergent security posture until this audit) and needed to be made explicit rather than left ambiguous.

**Options:**
1. **Delete `/api/v2/loads` now** — removes the audit surface immediately, but discards real completed migration work (domain/application layers) and the event-publishing capability, with no replacement plan.
2. **Promote v2 to be the live path, retire v1** — correct long-term direction, but requires migrating every remaining v1-only feature (load board browse/list/get, history, `markPickedUp`/`markDelivered` with BOL/POD photo gating, `claimLoad`'s One-Active-Load constraint pass-through, ratings) into the module's adapter layer, then cutting the frontend over — a multi-story effort, not something to do inside an audit-driven cleanup pass.
3. **Keep both, v1 authoritative, v2 secured but explicitly tracked as incomplete** — closes the immediate security gap (done, item 1), costs nothing extra today, and leaves the real migration decision to a future dedicated story rather than forcing it now.

**Recommendation:** Option 3.

**Decision:** Option 3, selected in this session per the Change Request Protocol's guidance that CODER-discovered architectural blockers not requiring immediate rework should be tracked and finished-as-is rather than triggering unplanned scope expansion. No LIBRARIAN persona was loaded in this session to make the call interactively, so per protocol §"If LIBRARIAN is unavailable" this is filed as the escalation artifact itself — the decision is recorded here rather than left as "BLOCKED: CHG-859" with no path forward, since Option 3 requires no rework of any existing story's inputs and is reversible.

**What This Means Going Forward:**
- `/api/v1/loads` remains the production API. Do not point new frontend work at `/api/v2/loads` without a dedicated migration story.
- `/api/v2/loads` keeps the authorization fix from PROJECT_AUDIT item 1 (role gating + ownership checks matching v1's pattern) as defense-in-depth, but is not to be treated as feature-complete or production-ready — it's missing load board browse/list, history, pickup/delivery document gating, and ratings.
- A future migration story (not yet created — no US-### assigned) is required before v1 can be deprecated or v2 promoted to the primary API. That story should scope: (a) porting the remaining `LoadService` features into `modules/load`'s adapter layer, (b) a frontend cutover to `/api/v2`, (c) a deprecation window for `/api/v1`, (d) deleting the legacy `controller.LoadController`/`service.LoadService`.
- `.claude/learnings.md` Technical Debt Ledger row for "Dual LoadController coexistence" updated to reference this ticket and correct the stale RLS claim.

**Next Steps:**
1. LIBRARIAN (when next loaded) reviews this decision and either ratifies it or reopens.
2. BA scopes the future v1→v2 migration story when there's a reason to prioritize it (not urgent — v2 is secured, not exposed to real risk).
3. No code changes beyond what item 1 already shipped.

**Status:** CHG-859 RESOLVED (Option 3 applied; no further action required until a migration story is scoped)
