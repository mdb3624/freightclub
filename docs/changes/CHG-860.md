## CHG-860: E2E CI Red Repo-Wide — `TestAuthController.deleteTestUser` 500s on Every Call (Tenant Context Lost Across OSIV-Shared Transactions)

**Discovered By:** REVIEWER phase of `/run-story` for US-861, which rejected PR #80 on a red `E2E Tests — Playwright` required check. Investigation showed the same check has been red/cancelled on the last 10 consecutive runs of `main`, unrelated to any specific PR's code.

**Root Cause:**

`TestAuthController.deleteTestUser` (`DELETE /api/test/auth/users/{id}`, the E2E per-test cleanup call, invoked after nearly every one of the ~125 E2E tests) called `userRepository.findById(userId)` and then `userRepository.save(user)` as two separate Spring-managed transactions. With `spring.jpa.open-in-view` at its Spring Boot default (`true`), both transactions shared one physical connection checked out once from `TenantAwareDataSource`. That class applies `SET LOCAL app.current_tenant` once per **connection checkout**, and `SET LOCAL` is transaction-scoped — it applied correctly for `findById()`'s transaction, then was reset at that transaction's commit. `save()`'s later `UPDATE` then ran with no tenant context bound; RLS's fail-closed `users_tenant_isolation` policy (V20260721_1402) filtered the row to zero matches instead of throwing, and Hibernate raised `StaleObjectStateException` (surfaced to the client as a generic 500) because it expected exactly one row updated.

This was latent until `freightclub_runtime` lost `BYPASSRLS` (US-857, 2026-07-21/23) — before that, RLS never actually filtered anything for this role regardless of `app.current_tenant`, so the missing context didn't matter.

**Blast Radius:** Every E2E test run on `main` and every open PR since US-857 landed — not specific to US-861's PR. Confirmed via `gh run list --branch main`: 10/10 recent runs cancelled or failed.

**Fix:** Added `@Transactional` to `deleteTestUser` so `findById()` and `save()` run as one transaction — the `SET LOCAL` that `TenantContextHolder.setTenantId()` re-issues onto an already-active transaction (documented behavior, see `TenantContextHolder`'s own javadoc) stays valid for both statements instead of being reset between them.

**Verification:** New regression test `AuthIntegrationTest.deleteTestUser_softDeletesUser_loginFailsAfterward` — confirmed red (500) against the pre-fix code, green after the fix. Full backend suite: 950/950, no regressions.

**Branch:** `fix/e2e-test-cleanup-tenant-context-reset` (off `main`, independent of `feature/US-861-notify-carrier-on-assignment`).

**Status:** CHG-860 — fix implemented and verified locally; PR pending.
