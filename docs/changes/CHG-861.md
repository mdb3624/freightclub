## CHG-861: E2E Tests Racing a Single-Use Rotating Refresh Token via the Shared `auth.json`

**Discovered By:** Follow-on investigation from CHG-860 — fixing the `deleteTestUser` 500 storm stopped that specific error but E2E CI still timed out. Root-caused during the same session.

**Root Cause (two related bugs):**

1. **Shared static session + single-use token rotation.** `RefreshTokenService.rotateRefreshToken()` revokes the old refresh token and issues a new one on every `/auth/refresh` call (single-use, by design). `AuthInitializer.tsx` unconditionally calls `refreshAccessToken()` on every mount — a deliberate security choice, since the access token is memory-only and never persisted (`authStore.ts`). Playwright's global default `storageState: 'auth.json'` (`playwright.config.ts`) hands the *same static cookie snapshot* to every test's browser context. The first test to mount rotates the shared token server-side; every other test sharing that snapshot then fails its own refresh call and gets logged out, redirected to the marketing home page — 100% reproducible for any test relying on the implicit global session, not flakiness.

2. **`TestDataSeeder`-based tests never applied the refresh cookie.** `TestDataSeeder.createTestUser()` captures a `refreshToken` from the register response's `Set-Cookie` header but several call sites (`smoke.spec.ts`, `hos-widget.spec.ts`) only wrote the (unused/dead) `freightclub_access_token` key to `localStorage` and never called `page.context().addCookies()` — so `AuthInitializer`'s refresh call had no cookie to work with at all, failing every time. A related ordering bug: seeding auth state via an initial `page.goto('/')` + `page.evaluate()` before navigating to the real target route is itself a second full page load, which also mounts `AuthInitializer` and consumes the single-use token before the real navigation gets to use it.

**Fix:** For files/`describe` blocks relying on the implicit shared session, gave each its own isolated per-test login (`test.use({ storageState: { cookies: [], origins: [] } })` + register-and-log-in-through-the-real-UI, matching the pattern already used successfully in 11 other spec files). For `TestDataSeeder`-based tests, added the missing `page.context().addCookies()` call and switched from `goto → evaluate → goto` to `addInitScript` + a single `goto`, so no wasted mount consumes the token before the real navigation.

**Files changed:** `frontend/e2e/design-system/US-842-layout-shell.spec.ts`, `frontend/e2e/design-system/US-843-shipper-dashboard.spec.ts`, `frontend/e2e/home-page.spec.ts`, `frontend/e2e/hos-widget.spec.ts`, `frontend/e2e/smoke.spec.ts`.

**Verification:** Local Docker test env, single-worker isolated runs per file:
- `smoke.spec.ts`: 7/7 passed
- `hos-widget.spec.ts`: 7/7 passed
- `home-page.spec.ts`: 12/12 passed
- `design-system/US-842-layout-shell.spec.ts`: 5/5 passed
- `design-system/US-843-shipper-dashboard.spec.ts`: 3/4 + 1 skipped passed in the "shipper dashboard reskin" block; the 5 tests in the sibling "US-843 adversarial" block (using the identical login helper) timed out waiting for the login redirect. No backend errors logged during the failure. Given the identical code path passes in the sibling block in the same file, this looks like local-machine resource contention from a long, heavy session (repeated Docker rebuilds, browser automation, video encoding) rather than a code defect — not independently confirmed against real CI in this session.

**Status:** CHG-861 — fix implemented and mostly verified locally (31/36 tests confirmed green in isolation); PR pending. The remaining 5-test cluster should be re-checked against real CI before being treated as fully resolved.
