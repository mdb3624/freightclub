# DB-002 — Timestamp Defaults on `loads` Table

**As an** engineer,
**I want** `loads.created_at` and `loads.updated_at` to have `DEFAULT CURRENT_TIMESTAMP` and `ON UPDATE CURRENT_TIMESTAMP`,
**so that** inserts without explicit timestamps don't fail and audit columns are always populated.

## Acceptance Criteria

- [ ] A Flyway migration adds `DEFAULT CURRENT_TIMESTAMP` to `loads.created_at`.
- [ ] The migration adds `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` to `loads.updated_at`.
- [ ] Existing rows with NULL `created_at` or `updated_at` are backfilled with a reasonable value (e.g., `NOW()` or a sentinel date).
- [ ] After migration, an `INSERT INTO loads (...)` without specifying `created_at` or `updated_at` succeeds and both columns are populated.
- [ ] The JPA entity (`Load.java`) is updated to remove any `@PrePersist` / `@PreUpdate` logic that manually sets these timestamps if the database default now handles it (or retained and verified consistent).
- [ ] Migration follows the Flyway naming convention.
