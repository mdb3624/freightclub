# DB-011 — Add `refresh_tokens.revoked_at` Column

**As an** engineer,
**I want** a `revoked_at DATETIME NULL` column on `refresh_tokens`,
**so that** token revocation events carry a timestamp for security auditing, not just a boolean flag.

## Acceptance Criteria

- [ ] A Flyway migration adds `revoked_at DATETIME NULL` to `refresh_tokens`.
- [ ] Existing rows where `revoked = 1` (or equivalent) are backfilled with a sentinel `revoked_at` value (e.g., the time of migration, or a configured past timestamp) to avoid NULL in historically revoked rows.
- [ ] The application's token revocation logic sets `revoked_at = NOW()` (instead of or in addition to the existing boolean flag).
- [ ] The existing `revoked` boolean column is retained for now (backward compatibility); both are updated on revocation.
- [ ] Security audit queries can now filter `WHERE revoked_at BETWEEN :start AND :end` to identify revocation activity over time.
- [ ] Migration follows the Flyway naming convention.
