# BUG-003 — Date Comparison Fix: Use `new Date()` Not String Ordering

**Phase:** 1.2
**Persona:** Shipper
**Status:** ✅ Built

## Story

As a shipper, I want cross-field date validation in the load form to compare actual Date objects so that the validation is correct regardless of datetime string format or locale.

## Background

`LoadForm.tsx` was comparing `datetime-local` input values as strings (e.g. `"2026-03-15T08:00" > "2026-03-14T18:00"`). String comparison on ISO-format dates happens to work in most cases but is brittle — non-ISO formats or locale-specific formatting silently breaks the ordering check.

## Acceptance Criteria

- All cross-field date comparisons in `LoadForm.tsx` use `new Date(value)` before comparing
- Zod schema date refinements use `new Date()` for all comparisons
- Invalid date windows (e.g. latest pickup before earliest pickup) are caught and displayed as inline errors
- Existing valid date windows continue to pass validation without error
