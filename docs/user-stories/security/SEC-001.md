# SEC-001 — Race Condition Fix: Load Claiming

**Phase:** 1.2
**Persona:** Platform
**Status:** ✅ Built

## Story

As the platform, I want load claiming to acquire a row-level lock before checking status so that two truckers cannot simultaneously claim the same load.

## Background

Under concurrent load, two truckers can both read `status = OPEN` before either write completes. Both writes succeed; both truckers believe they own the load. This is a data integrity failure in the core transaction of the platform.

## Acceptance Criteria

- `LoadService.claimLoad()` acquires `LockModeType.PESSIMISTIC_WRITE` on the `loads` row before checking status
- A second claim request for the same load while the first is in-flight receives a 409 Conflict response
- A unique partial index on `(load_id) WHERE status = 'CLAIMED'` is added as a secondary DB-level safety net
- A concurrency integration test (two simultaneous claim requests) passes and confirms only one succeeds
