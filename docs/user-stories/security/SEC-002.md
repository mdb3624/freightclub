# SEC-002 — Race Condition Fix: Refresh Token Rotation

**Phase:** 1.2
**Persona:** Platform
**Status:** ✅ Built

## Story

As the platform, I want refresh token rotation to acquire a row-level lock on the token row so that two simultaneous refresh requests cannot both receive valid new tokens from the same refresh token.

## Background

Two concurrent `/auth/refresh` calls can both read `revoked = false` before either marks the token as revoked. Both receive a new access token, creating a token duplication window that could be exploited if a token is stolen.

## Acceptance Criteria

- `RefreshTokenService.rotate()` acquires `LockModeType.PESSIMISTIC_WRITE` on the refresh token row before checking revocation status
- The second concurrent request receives a 401 Unauthorized response
- A unit test covering the rotation race condition passes
